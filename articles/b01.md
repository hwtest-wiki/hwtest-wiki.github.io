# 如何正确提交一个测试 Bug：从发现到关闭，硬件测试的完整闭环

<div class="article-meta"><span class="badge">📘 实战番外</span><span class="badge">⏱ 约 18 分钟</span><span class="badge qr">🔧 含工程师速查</span></div>


先看两张 Bug 单。

**A 单**，标题写着"**板子有问题，偶尔会重启**"，正文一句话："测试的时候发现会重启，帮忙看下。" 开发拿到手一脸懵：哪块板？什么条件？多大概率？重启前什么现象？于是回一句"**本地复现不了，麻烦补充信息**"，状态打回。你补一点、他问一点，来回拉扯，**一周过去了 Bug 还挂在那**。

**B 单**，标题写着"**高温 60℃ 下，A 板约 1/5 概率开机重启，重启瞬间 Vcore 掉到 0.7V**"，正文里板号、批次、固件版本、复现步骤、复现率、示波器抓的掉压波形、供电条件一应俱全，还附了一句"初步观察：重启前 Vcore 有明显跌落，怀疑上电时序或负载瞬态"。开发照着条件一摆，**半小时就复现、定位、给出修改方向**。

**同一个 Bug，两种命运。差别不在这个 Bug 有多难，而在你怎么提。** 这一篇要讲的，就是怎么让你每一张单都是 B 单。

我们用一个**你大概率碰到过的典型场景**贯穿全程：**某块板在高温下，开机有一定概率起不来、直接重启。** 跟着这个案例，把整个闭环走一遍。

**读完这篇，你能搞懂：**

- ✅ 发现 Bug 后**先别急着提**——复现、隔离、排除自己，三步自查怎么做；
- ✅ 硬件 Bug 的**证据链**到底要收哪些（板号 / 批次 / 固件 / 条件 / 复现率 / 客观证据）；
- ✅ **严重度（Severity）和优先级（Priority）**有什么区别，别再混着填；
- ✅ 一张**开发一眼就懂**的 Bug 单长什么样——标题公式 + 正文模板；
- ✅ 提交之后怎么**跟踪、回归验证、关闭**，以及跟开发沟通怎么"对事不对人"。

## 全流程一张图：发现 → 处理 → 提交 → 跟踪 → 关闭

先建立整体框架。一个测试 Bug 从冒头到彻底了结，一共五个阶段：

1. **发现**——测出异常现象；
2. **处理**——自己先把它搞清楚（复现、隔离、排除自己、收证据）；
3. **提交**——写成一张开发看得懂的 Bug 单，进缺陷系统、指派对人；
4. **跟踪**——推动修复、回归验证；
5. **关闭**——确认根因修好、沉淀防复发。

很多人把"发现"和"提交"直接连起来——**测到异常，立刻甩一张单出去**。这正是烂单的根源。真正拉开专业度的，是中间那个**"处理"**环节：你在提交之前，替开发把能做的功课先做了。

> **一句话记住：Bug 的生命周期是"发现→处理→提交→跟踪→关闭"五步，别把"发现"直接跳到"提交"。中间的"处理"（复现+隔离+排除自己+收证据）做得越足，后面跟踪、关闭越省事。测试的价值是"推动 Bug 被修掉"，不是"甩出一个 Bug"。**

## 一、先别急着提：三步自查，别当"误报侠"

测到高温下会重启，手别急着按提交键。**先过三关**，这三关决定了你这张单是"金子"还是"噪声"。

**第一关 · 能不能稳定复现？** 偶发问题最坑。你得先把它变成一个**可复现**的现象：在什么条件下、按什么步骤、多大概率会出。高温重启这个例子，你要做的是——固定温箱到 60℃，重复开机 20 次，记下重启了几次（比如 4 次，复现率约 1/5）。**"复现率"是硬件 Bug 的关键数据**：必现的和万分之一才出一次的，处理优先级天差地别。连你自己都复现不了的，先别提，容易变成谁都查不动的悬案。

**第二关 · 是硬件问题还是软件问题？** 用**二分法**快速定位：换一块同批次的板子还复现吗？（缩小到"是不是个体差异"）换一个旧版本固件还复现吗？（缩小到"是不是软件引入的"）换一套电源 / 线缆 / 治具还复现吗？（缩小到"是不是外围环境"）。这一步不是要你查出根因，而是**把范围收窄，好让单子指给对的人**——硬件问题甩给软件、软件问题甩给硬件，都是在浪费所有人的时间。

**第三关 · 排除你自己。** 提单之前，诚实地问一遍：是不是我操作错了？线接反了、供电电压设错了、治具没插紧？是不是环境问题——ESD、温度没到位、静电、接地不良？是不是测试脚本 / 上位机自己的 Bug？**"误报"提多了，你的单子会开始被开发默认打折扣**，这是测试工程师最不该丢的东西——**可信度**。

三关都过了，这才是一个**值得提**的 Bug。

> **一句话记住：提交前先过三关——①能不能稳定复现（并量出复现率，如 20 次里 4 次）；②硬件还是软件（换板/换固件/换环境三个二分法收窄范围）；③排除你自己（操作、接线、供电、环境、脚本）。三关不过就提，轻则被打回，重则拉低你单子的可信度。**

## 二、硬件 Bug 的命门：证据链（这是跟软件 Bug 最大的不同）

纯软件 Bug，开发常常自己就能在机器上复现；可**硬件 Bug 换一块板、换一个环境，现象可能就变了**。所以硬件测试 Bug 单最值钱、也最容易漏的，就是**证据链**——你要让开发**不在现场，也能还原你看到的一切**。

一张合格的硬件 Bug 单，证据链至少包含这几类：

- **身份信息**：板号 / 序列号、**批次（lot）**、硬件版本（rev）、固件 / BIOS 版本、上位机软件版本。同一个问题，A3 批次有、A4 批次没有，这条线索价值千金。
- **复现条件**：温度、输入电压、负载、时序、具体操作步骤——**越精确越好**。"高温下"不算条件，"温箱 60℃、输入 12V、满载"才算。
- **复现率**：20 次里 4 次，还是 100 次必现。这直接决定开发排查的方式和优先级。
- **客观证据**（硬件 Bug 的灵魂）：**示波器 / 逻辑分析仪抓的波形截图**、万用表读数、电源纪录、**热像图**、现象照片 / 录像、完整 log。高温重启这个例子，一张"重启瞬间 Vcore 从 1.0V 跌到 0.7V"的波形，胜过一千句"感觉是电源问题"。
- **首次出现 / 变化点**：什么时候开始出现的？换了什么之后开始的？（换了批次？改了固件？动了 layout？）

**硬件 Bug 单被打回最常见的原因，就是条件写不清——开发在他桌上复现不出来，只能反手一句"无法复现"。** 你把条件和证据一次性给足，就把这个来回彻底省掉了。

> **一句话记住：硬件 Bug 换板、换环境现象就变，所以证据链是命门——身份信息（板号/批次/rev/固件版本）+ 复现条件（温度/电压/负载/时序，要具体到数字）+ 复现率 + 客观证据（波形截图/读数/热像/照片/log）+ 变化点。让开发不在现场也能还原你看到的一切，就没有"无法复现"的打回。**

## 三、严重度 ≠ 优先级：两个字段别再混着填

缺陷系统里几乎都有两个字段，很多人分不清，随手一填：**严重度（Severity）**和**优先级（Priority）**。

- **严重度**：这个 Bug 造成的**后果有多严重**——是死机 / 烧板（致命）、还是功能不可用（严重）、还是偶发小问题（一般）、还是界面文字错别字（轻微）。这是**技术属性**，由现象本身决定。
- **优先级**：这个 Bug **要多急着修**——马上、这个版本、下个版本、有空再说。这是**项目属性**，由发布计划、影响范围一起决定。

两者**经常不一致**，这正是要分开填的原因：

- 一个**必死机**但只在实验室极端条件出现、量产用不到的问题——**严重度高，优先级未必高**。
- 一个只是**开机 logo 偏了几个像素**、但**量产每台都有、马上要出货**的问题——**严重度低，优先级却很高**。

高温概率重启这个案例：现象是重启（严重度偏高），但如果产品明确要在 60℃ 环境长期工作，那它**优先级也高**；如果只是超出规格的极限拷机才偶发，优先级就得往下压。**把这两个字段填对，是在帮项目做正确的排期，而不是每个 Bug 都喊"最高优先级"——都最高，就等于都不最高。**

> **一句话记住：严重度（Severity）= 后果多严重（技术属性，看现象），优先级（Priority）= 多急着修（项目属性，看发布计划和影响面）。两者常不一致：极端条件才出的死机可能"严重但不紧急"，量产必现的小瑕疵可能"不严重但很紧急"。别把每个单都填最高优先级。**

## 四、写一张开发一眼就懂的 Bug 单：标题公式 + 正文模板

功课做完了，现在把它写成单。一张好单，**标题让人秒懂、正文让人能复现**。

**标题公式 =【条件】+【现象】+【关键数据】。** 对比一下：

- ❌ "板子会重启"——没条件、没数据，开发得来问你三轮。
- ✅ "**高温 60℃ 下，A 板约 1/5 概率开机重启，重启瞬间 Vcore 掉到 0.7V**"——条件、现象、关键数据一句话齐了，开发扫一眼就知道是什么、有多严重、大概往哪查。

**正文用固定模板**，一项都别少：

1. **环境 / 配置**：板号、批次、硬件 rev、固件版本、上位机版本、测试设备。
2. **复现步骤**：一步一步，能照着做出来。编号列清楚，别写"按正常流程操作"。
3. **预期结果**：应该是什么样（开机正常进系统）。
4. **实际结果**：实际是什么样（约 1/5 概率重启）。
5. **复现率**：20 次里 4 次。
6. **证据附件**：波形截图、log、照片、录像。
7. **初步分析（可选，但别越界）**：你观察到的线索（"重启前 Vcore 有跌落，怀疑上电时序或负载瞬态"）。**注意：给线索、给方向，别下"就是你芯片坏了"这种结论**——定位是开发的活，你越界下结论，一旦错了反而制造对立。

**"预期 vs 实际"这一对，是 Bug 单的骨架**——它把"这是个 Bug"这件事讲成了客观事实，而不是你的主观判断。

> **一句话记住：标题 =【条件】+【现象】+【关键数据】一句话让人秒懂；正文用固定模板七项（环境/复现步骤/预期/实际/复现率/证据/初步分析）。"预期 vs 实际"是骨架，把 Bug 讲成客观事实。初步分析只给线索和方向，别越界替开发下"就是你的问题"这种结论。**

## 五、提交之后：跟踪、回归验证、对事不对人

单子提出去，不是甩锅结束，而是**你要一路推到它被真正修好**。

**进对系统、指派对人。** 用团队的缺陷管理系统（Jira、Azure DevOps、禅道、Bugzilla 等），别用微信 / 口头——**没进系统的 Bug 等于不存在**，没法追踪、没法统计、离职就丢。指派给对的模块负责人，指错人只会让单子在几个人之间踢皮球。

**看懂状态流转。** 一张单的典型生命周期：**新建 → 已确认 → 已分配 → 修复中 → 待验证 → 回归验证 → 关闭**（或打回重开）。你要盯着它别卡在某一环——比如卡在"待验证"好几天没人回归，那就是该你出手的时候。

**回归验证是测试的收尾硬活。** 开发说"修好了"**不等于**真修好了。你要**在原来的复现条件下重新测**，而且不能只测一次就算——概率性问题尤其要**多测几轮**（原来 20 次 4 次，那你至少再跑 20 次确认 0 次）。还要顺手**测一下边界和相邻功能**，防止"按下葫芦浮起瓢"——修了这个又碰坏那个。**只有你亲手复测通过，这个 Bug 才有资格进入关闭。**

**沟通对事不对人。** Bug 单是对着**问题**的，不是对着**人**的。多附客观证据（波形、数据、log），少下主观断语（"你这设计就是烂"）。一句"我这边在这个条件下观察到这个现象，数据在附件，帮忙一起看看"，比"你的芯片有问题"能让事情快十倍。**测试和开发是一队的，共同的敌人是 Bug，不是彼此。**

> **一句话记住：提交后要推到修好——①进缺陷系统（没进系统=不存在）、指派对人；②盯状态流转（新建→确认→分配→修复→待验证→回归→关闭），别让它卡住；③回归验证亲手在原条件下多测几轮 + 测边界，"开发说修好"≠真修好；④沟通对事不对人，附证据不下断语，测试和开发共同的敌人是 Bug。**

## 六、关闭与沉淀：让同一个坑，别有人再踩

回归通过，可以关闭了。但**专业的收尾，不止是点一下"关闭"**。

- **确认根因写清楚**：这个 Bug 到底是什么原因、怎么修的，写进单子。一句"已修复"关掉，等于把这次踩坑的经验也一起埋了。高温重启如果最终定位是"上电时序裕量不足，高温下某路电压建立偏慢"，那就把它记下来。
- **更新测试用例 / 检查清单**：把这个 Bug 变成一条**新的测试项**——"高温 60℃ 开机拷机 ×50 次"。这样下一块板、下一个项目，这个坑会被**自动拦住**，而不是靠人记得。**一个被沉淀成用例的 Bug，才真正产生了长期价值。**
- **横向同步**：同批次、同平台的其它板子，会不会有同样问题？主动排查一圈，比等它一个个爆出来强。

到这里，一个 Bug 才算真正**闭环**：不只是修好了这一次，而是让同一个坑，团队里不会再有人掉进去。

> **一句话记住：关闭 ≠ 点一下"关闭"。要①把根因和修复方式写清（别用"已修复"一笔带过）；②把这个 Bug 沉淀成一条新测试用例/检查项（如"高温60℃开机×50"），让坑被自动拦住；③横向排查同批次/同平台是否有同样问题。被沉淀成用例的 Bug，才产生长期价值。**

## 写在最后

回到开头那两张单。A 单和 B 单，测到的是同一个现象，差别全在**你在"提交"之前，替开发和项目多做了多少功课**：把偶发变成可复现、把"感觉"变成证据链、把"很严重"拆成严重度和优先级、把一句话变成能照做的模板、再一路盯到回归关闭、沉淀成用例。

**测试工程师的专业度，一半在于你能测出多刁钻的问题，另一半就在于你能把问题多干净利落地交出去、推着它闭环。** 会测，还要会提——这两件事加起来，才是一个让开发信任、让项目省心的测试。

这条**实战番外**支线，以后会不定期加餐，专门聊这些"技术之外、但同样决定你专不专业"的实战功夫。

## 🔧 工程师速查

### Bug 生命周期五步
**发现 → 处理 → 提交 → 跟踪 → 关闭**。拉开专业度的是中间"处理"（复现 + 隔离 + 排除自己 + 收证据），别从"发现"直接跳"提交"。

### 提交前三步自查
| 关 | 要做什么 | 不过怎么办 |
|---|---|---|
| ① 复现 | 固定条件重复，量出**复现率**（如 20 次 4 次） | 复现不了先别提 |
| ② 隔离 | 换板 / 换固件 / 换环境三个二分法，收窄硬件还是软件 | 指错人 = 踢皮球 |
| ③ 排除自己 | 操作 / 接线 / 供电 / 环境 / 脚本 | 误报多了掉可信度 |

### 硬件证据链（必收五类）
1. **身份**：板号 / 序列号 / 批次 lot / 硬件 rev / 固件·BIOS 版本 / 上位机版本
2. **条件**：温度 / 电压 / 负载 / 时序（要具体数字，"60℃、12V、满载"）
3. **复现率**：20 次里 4 次 / 100 次必现
4. **客观证据**：示波器波形截图 / 万用表读数 / 热像 / 照片·录像 / 完整 log
5. **变化点**：换了批次 / 改了固件 / 动了 layout 之后开始的

### 严重度 vs 优先级
| | 严重度 Severity | 优先级 Priority |
|---|---|---|
| 看什么 | 后果多严重（技术属性） | 多急着修（项目属性） |
| 谁决定 | 现象本身 | 发布计划 + 影响面 |
| 例 | 极限条件偶发死机=严重但可不紧急 | 量产 logo 偏移=不严重但很紧急 |

**别把每个单都填最高优先级——都最高等于都不最高。**

### Bug 单标题公式
**【条件】+【现象】+【关键数据】**
✅ "高温 60℃ 下，A 板约 1/5 概率开机重启，重启瞬间 Vcore 掉到 0.7V"
❌ "板子会重启"

### Bug 单正文模板（七项）
1. 环境 / 配置（板号·批次·rev·固件·上位机·设备）
2. 复现步骤（编号，能照做）
3. 预期结果
4. 实际结果
5. 复现率
6. 证据附件（波形 / log / 照片）
7. 初步分析（给线索给方向，**别越界下结论**）
> "预期 vs 实际"是骨架，把 Bug 讲成客观事实。

### 状态流转与回归
**新建 → 已确认 → 已分配 → 修复中 → 待验证 → 回归验证 → 关闭**（可打回重开）
- 进缺陷系统（Jira / Azure DevOps / 禅道 / Bugzilla），**没进系统 = 不存在**
- 回归验证：**在原条件下亲手复测**，概率性问题多测几轮（原 20/4 → 再跑 20 次 0 次）+ 测边界，防"按下葫芦浮起瓢"
- "开发说修好" ≠ 真修好

### 关闭与沉淀
- 根因 + 修复方式写清（别只写"已修复"）
- 沉淀成**新测试用例 / 检查项**（如"高温 60℃ 开机 ×50"），让坑被自动拦住
- 横向排查同批次 / 同平台

### 沟通铁律
对事不对人：**多附客观证据，少下主观断语**。测试和开发共同的敌人是 Bug，不是彼此。

### 常见翻车
- 只写"有问题 / 会重启"，无条件无数据 → 必被打回
- 硬件问题甩软件、软件问题甩硬件 → 全组浪费时间
- 复现率不写 → 开发不知道该怎么查
- 越界下结论"你芯片就是坏的" → 错了制造对立
- 修完不复测就关 → 假关闭，现场再爆
