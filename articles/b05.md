# 失效分析与故障定位：板子"偶尔"会重启，怎么揪出真凶？

<div class="article-meta"><span class="badge">📘 实战番外</span><span class="badge">⏱ 约 20 分钟</span><span class="badge qr">🔧 含工程师速查</span></div>


一块用了很久的板子，测试同事报了个让人头皮发麻的 Bug：**"偶尔会重启。"**

多偶尔？开机十次大概有一两次，跑着跑着毫无征兆黑一下、又自己起来了。你拿过来一测——正常。反复开关机几十次——正常。示波器挂上去盯了半小时——正常。你几乎要在报告上写"无法复现（NFF, No Fault Found）"了，结果一还给产线，第二天又报三台。

这就是硬件测试里最磨人的一类问题：**偶发故障（intermittent fault）**。它不像"一上电就冒烟"那样痛快，它躲着你。很多人对付它的方式是"换件大法"——先换电源 IC，不行换电容，再不行换整板，换到不复现为止。运气好蒙对一次，运气不好换了一圈钱花了、时间没了，根因还在原地笑你。

**换件不是定位，蒙对不是找到。** 真正的失效分析是一套可复现、可追溯的方法：不管故障多偶发、现象多诡异，你都能顺着一条逻辑链，把范围一步步收窄，最后指着一个点说——"就是它，我能让它复现，也能让它消失。"

这一篇，我们拿一块 **TPS54331 降压电源板"偶发欠压复位"** 当靶子，把这套方法从头走一遍。案例只是载体，方法本身跟这颗芯片没关系；学会了，不管是电源打嗝、通信偶发误码，还是接口偶尔枚举失败，你都能照着套。

读完这篇，你能带走：

- ✅ 一套**六步故障定位法**：复现 → 二分 → 假设 → 隔离 → 确认 → 防再发，一条链走到底
- ✅ 偶发故障怎么**"提高复现率"**——把躲着你的问题逼出来，别再写"无法复现"
- ✅ **二分定位（半分法）**：怎么用最少的实验，最快把范围砍到最小
- ✅ **鱼骨图 6M + 5 Why + 故障树**这几件经典武器，到底怎么用在硬件上，而不是画个图交差
- ✅ **根因确认的铁律**：为什么"能复现 + 能消除"（能开能关）才算真找到
- ✅ 一份 **8D 报告 Excel 模板**（文末下载），把整个定位过程沉淀成可追溯、可防再发的闭环

![图1：偶发故障——开机大多正常，只在某些条件下电压突然跌一下触发复位，测的时候又抓不到，最磨人的一类 Bug](/images/b05/ill01_intermittent.png)

## 一、先改心态：偶发故障不是靠运气，是靠方法

面对"偶尔会重启"，两种工程师会走向完全不同的结局。

**A 类工程师**：凭经验猜。"电源 IC 老化了吧"→ 换一颗；"电容不行吧"→ 换一批；"要不整板重做"。每换一次都要等复现来验证，而偶发故障可能几小时才现一次，一轮下来一天没了。更糟的是，就算换完不复现了，他也说不清到底是哪个改动起了作用——**这不叫定位，这叫拿概率赌运气。**

**B 类工程师**：把它当成一个"信息不足"的问题。他知道偶发故障之所以难，是因为**触发条件还没被摸清**。于是他不急着动手换件，而是先问：这故障在什么条件下更容易出？能不能想办法让它**从"偶尔"变成"每次"**？一旦能稳定复现，偶发故障就退化成了一个普通故障——剩下的就是顺着逻辑链往下挖。

这一篇讲的就是 B 类工程师的套路。它可以拆成六步（图2）：

![图2：六步故障定位法总览——复现 → 二分 → 假设 → 隔离 → 确认 → 防再发，一条逻辑链，不靠蒙](/images/b05/ill02_sixstep.png)

1. **复现**：把偶发逼成常发，先拿到一个能稳定观测的现象
2. **二分**：用半分法把"问题在哪一段"快速砍到最小范围
3. **假设**：在缩小的范围里，用鱼骨图 6M + 5 Why 列出所有可能原因
4. **隔离**：一次只动一个变量，逐个验证 / 排除假设
5. **确认**：找到能**同时**"制造"和"消除"故障的那个因子——那才是根因
6. **防再发**：纠正 + 验证 + 把教训固化成规范，用 8D 闭环收口

下面一步步走。

## 二、第一步 · 复现：把"偶尔"逼成"每次"

**定位的前提是能观测，观测的前提是能复现。** 一个几小时才出一次的故障，你没法调试它——你需要先把它的复现率拉高。

怎么拉高？核心思路是：**沿着"它更容易在什么条件下出现"这条线，主动施加压力（stress），把系统推到临界点附近。** 常用的几根"加压杠杆"：

- **加负载 / 加负载瞬态**：电源类问题，满载、动态突加突卸负载往往是照妖镜。让 MCU 跑满、外设同时开，或用电子负载做阶跃。
- **加温度应力**：热风枪吹、冷冻剂（freeze spray）喷、或整板进温箱做高低温循环。**很多偶发故障是温度相关的**——低温下电容容值/电感感值变化、限流点漂移，高温下结温逼近保护阈值。
- **拉偏供电**：把输入电压拉到规格下限（甚至略低），电源的裕度最小、最容易露馅。
- **加时间 / 加次数**：写脚本让它反复上下电、反复负载切换几千次，把"万分之一"的概率在一晚上里跑出来（这一步正好可以接番外04 的**自动化测试**——让机器整夜帮你复现）。
- **加扰动**：敲击、弯板（找虚焊 / 微裂）、插拔线缆——机械应力能揪出接触类偶发。

![图3：提高复现率——加负载、加温度、拉偏供电、加次数、加扰动，把躲着你的偶发故障逼到临界点现形](/images/b05/ill03_reproduce.png)

**TPS54331 案例**：我们先把现象量化（沿用番外01 的 5W2H 思路）——什么现象（输出掉、MCU 复位）、什么时候（负载切换的瞬间偏多）、多频繁（十次一两次）。据此施压：**用电子负载对 3.3V 输出做突加突卸的负载瞬态，同时把环境降到低温**。结果复现率从"十次一两次"直接飙到"几乎每次负载突加都跌一下触发复位"。

到这一步，最大的敌人——"抓不到"——已经被打败了。偶发故障变成了常发故障。

> ⚠️ **别把测量本身当成故障。** 提高复现率的同时，要保证你的观测手段是可信的：示波器带宽/采样率够不够（漏抓窄脉冲）、探头地线环路会不会引入伪振铃、触发设错会不会误判。**测不准，后面全白搭**——这一条本身就是鱼骨图里"测量（Measurement）"那根大骨，后面还会提。

## 三、第二步 · 二分：半分法，用最少实验砍最大范围

能稳定复现之后，别急着怀疑某个具体元件。先做一件更聪明的事：**把系统一刀砍两半，判断故障在哪一半。** 这就是二分定位（半分法 / divide and conquer），它是所有高效 debug 的核心——每做一次判断，就把嫌疑范围砍掉一半。

硬件上怎么砍？沿着**信号链 / 供电链**分段，在中间设一个观测点或切割点：

![图4：二分定位——沿供电链/信号链分段，每次一刀砍两半，判断故障落在哪一段，范围指数级收窄](/images/b05/ill04_bisect.png)

**TPS54331 案例**的供电与复位链大致是：`输入 12V → 输入电容/EMI → TPS54331 → 输出 LC 滤波 → 3.3V → MCU 供电脚 → MCU 欠压复位（BOR）`。

我们在负载瞬态触发复位的那一刻，同时看几个点：

- **先砍一刀问"是电源跌了，还是 MCU 自己的问题？"** 用示波器抓 3.3V 输出：如果事件瞬间 **VOUT 明显跌落到 MCU 的欠压复位阈值以下**，那问题在电源侧，MCU 只是受害者——右半段（MCU）洗清嫌疑，砍掉。
- **再在电源侧砍一刀问"是输入垮了，还是 IC/输出侧的问题？"** 同时看输入 12V：如果输入稳如泰山、只有输出在跌，那问题锁定在 `TPS54331 + 输出 LC + 负载` 这一段——输入侧砍掉。

两刀下去，范围就从"整块板"收窄到了"降压级本身在负载瞬态下 hold 不住输出"。**注意：到这里我们一个元件都还没换**，纯靠观测点 + 二分，就把战场缩小了一个数量级。这就是方法论比"换件大法"高效的地方。

## 四、第三步 · 假设：鱼骨图 6M + 5 Why，把可能原因摊开

范围缩小了，接下来要**穷举**这段范围内所有可能的原因——不是挑一个最顺眼的先怀疑，而是先摊开，别漏。两件经典武器：鱼骨图负责"横向铺全"，5 Why 负责"纵向挖深"。

### 4.1 鱼骨图（Ishikawa / 因果图）：6M 横向铺全

鱼骨图把"结果"（这里是"负载瞬态下输出跌落触发复位"）放在鱼头，用六根大骨（**6M**）系统地分类所有可能原因，逼你各个方向都想到，不留死角：

![图5：鱼骨图 6M——人机料法测环六个方向系统铺开可能原因，逼你别漏，是穷举假设的框架](/images/b05/ill05_fishbone.png)

- **Material 料**：输出电容 DC bias 下容值衰减、电感饱和电流不足（近饱和感值骤降）、输入电容偏小、某批次来料参数漂。
- **Machine 机（设备/器件本身）**：TPS54331 补偿不足导致瞬态欠冲、峰值限流点偏低、这颗 IC 个体偏弱。
- **Method 法（设计/工艺方法）**：反馈/补偿网络布局差、地平面回流路径长、软启动/环路带宽设定不当、焊接工艺导致虚焊。
- **Measurement 测**：探头带宽不足漏抓窄跌落、地线环路引入伪信号、触发阈值设错导致误判——**先排除"是不是我测错了"**。
- **Environment 环**：低温下容值/限流/感值变化、高温结温逼近保护、输入源内阻大在瞬态下被拉低。
- **Man 人**：装错料值、贴错料、漏焊、批次混料。

一次把六个方向全列出来，你就不会犯"一口咬定是电容、结果是限流"的隧道视野错误。

### 4.2 5 Why：顺着一条线纵向挖深

鱼骨列出的是"候选"，5 Why 则是把最可疑的那条线一路问到底，问到**再问下去就是要改设计/改流程**为止：

![图6：5 Why——从"MCU 复位"一路追问到"负载瞬态峰值电流触发限流+低温使裕度更小"，挖到可动手的根因层](/images/b05/ill06_5why.png)

**TPS54331 案例的 5 Why：**

1. **为什么 MCU 会复位？** → 因为它的供电 3.3V 在某个瞬间跌到了欠压复位阈值以下。
2. **为什么 3.3V 会瞬间跌？** → 因为负载突加的瞬间，电源没能及时把输出撑住，出现了大的电压欠冲（undershoot）。
3. **为什么撑不住、欠冲这么大？** → 因为负载瞬态的峰值电流触发了 TPS54331 的**逐周期峰值限流**，输出电流被"削顶"，供不上突增的负载。
4. **为什么峰值电流会触到限流？** → 因为限流点本身裕度就不大，而**低温下限流阈值下移、电感也更接近饱和**，一叠加，瞬态尖峰就越过了限流线。
5. **为什么裕度这么小？** → 因为设计选型时按稳态满载算的限流裕度，**没有把"负载瞬态尖峰 + 低温漂移"这两个动态/环境因子叠进去算**。

到第 5 层，我们挖到了一个**可以动手的根因层**：不是"某个元件坏了"，而是"限流裕度在动态+低温工况下不够"。这比"换个电容试试"精准太多了。

> 💡 5 Why 不一定正好 5 层，核心是"问到能落到具体、可纠正的动作为止"。而且它常常不是单线——鱼骨图里几条骨可能同时贡献（低温 + 瞬态 + 电感近饱和三者叠加），这时用**故障树（FTA）**更合适：把"输出跌落"当顶事件，往下用"或门 / 与门"把各基本原因的组合关系画清楚，看清到底是"任一条件就触发"还是"几个条件叠加才触发"。

## 五、第四步 · 隔离：一次只动一个变量

假设列了一堆，怎么确认到底是哪个？铁律只有一条：**一次只改一个变量（one factor at a time），改完就回来看现象变没变。** 一次动两个，故障消失了你都不知道是哪个的功劳——这是新手最常犯、也最浪费时间的错。

几件趁手的隔离工具：

![图7：隔离变量——冷冻剂/热风枪局部激发定位温度敏感点、单因子替换、示波器抓瞬态，一次只动一个](/images/b05/ill07_isolate.png)

- **冷冻剂 / 热风枪局部激发**：这是定位温度相关偶发故障的神器。用**窄嘴冷冻剂**只喷某一个元件（比如电感、或电源 IC），如果一喷它复现率就变、别的不变，那这个元件就是温度敏感的嫌疑点；反过来用**热风枪**局部加热同理。**这一步能把"整板温度相关"缩小到"具体哪颗器件温度相关"。**
- **单因子替换**：把输出电容临时加大一档（只改这一个）——如果欠冲明显变小、复现率下降，说明"输出储能/环路"这条线有份；换一颗限流裕度更大的同族电源 IC（只改这一个）——如果直接不复现，限流假设坐实。**每次只换一个，换完记录，再换回来复验。**
- **示波器抓瞬态真相**：在负载突加那一刻，同步抓 **VOUT + SW 开关节点 + 电感电流**。如果能看到电感电流尖峰"顶"在某个值上被削平、同时 VOUT 往下掉——那就是限流触发的铁证，假设直接被波形实锤。

**TPS54331 案例**：冷冻剂喷电感和 IC，发现**低温方向复现率显著上升**（证实温度相关，且指向限流/电感）；把输出电容加大一档，欠冲变浅、复现率下降（证实储能/环路有份但不是全部）；最后临时换一颗限流裕度更大的方案，负载瞬态下**不再触发限流、VOUT 不再跌破阈值、复位消失**。波形上也清清楚楚看到之前那个"被削平的电感电流尖峰"没了。

## 六、第五步 · 确认：能复现 + 能消除，才算真凶

这是整套方法论里**最容易被跳过、也最关键**的一步。很多人做到"改了个东西不复现了"就收工——但"不复现"可能只是运气，也可能是你顺手改的别的东西起了作用。

**根因确认的铁律：一个原因，只有当你既能用它"制造"故障、又能用它"消除"故障，才算被确认。俗称"能开能关"。**

![图8：根因确认铁律——能主动"制造"故障、又能主动"消除"故障（能开能关），双向可控才算真凶，否则只是相关不是因果](/images/b05/ill08_rootcause.png)

具体到案例，怎么"能开能关"：

- **能关（消除）**：提高限流裕度（换裕度更大的方案 / 加大输出电容 / 优化补偿）→ 负载瞬态下不再触发限流 → 复位消失。✅
- **能开（制造）**：把改动撤回、或反过来故意把限流裕度调得更小、把温度压得更低 → 复位**必然重现**。✅

两个方向都可控，因果关系就闭环了——**"负载瞬态峰值电流在低温下越过限流阈值，导致输出欠压触发 MCU 复位"** 这个根因，实锤。这时候你在报告里写的就不是"疑似"，而是"经双向验证确认"。

> ⚠️ **警惕"相关 ≠ 因果"。** 你换了个电容不复现了，可能只是新电容的 ESR/容值恰好把工作点推离了临界，根因其实还是限流裕度——下批来料参数一漂，它又回来了。只有能双向控制的，才是真根因；否则你修的是症状，不是病。

## 七、第六步 · 防再发：8D 闭环，把这次的血汗变成规范

找到根因、改好了，任务还没完。**一次失效分析真正的价值，是让"同一个坑"和"同一类坑"再也不发生。** 业界最通用的收口框架是 **8D（Eight Disciplines，八个步骤）**——它把从接到问题到彻底关闭的全过程标准化，也正好和番外01 的"Bug 全生命周期"接上（01 讲怎么报、这里讲怎么系统性解决和防再发）。

![图9：8D 闭环 D0–D8——从应急遏制到根因、纠正、防再发、闭环，把一次故障定位沉淀为可追溯、防复发的标准动作](/images/b05/ill09_8d.png)

**8D 八步（D0–D8）：**

- **D0 应急反应**：先止血。偶发复位若影响出货，先加临时遏制（如产线降额筛选、暂缓发货），保护客户，再慢慢查。
- **D1 组队**：拉上硬件、测试、工艺、采购——限流+低温+电感这种问题，往往一个人视角不全。
- **D2 描述问题**：用 5W2H 把现象量化写死（什么条件、多频繁、影响什么）——这一步就是番外01 教的"好 Bug 长什么样"。
- **D3 临时遏制（ICA）**：在根因修复前，先上可验证的临时措施（如软件加复位重试、产线加低温复测），并验证它确实挡住了问题。
- **D4 找根因并验证**：本篇二~六步的全部——复现、二分、假设、隔离、**双向确认**。8D 的灵魂就在这一步，前面那套方法就是为它服务的。
- **D5 选并验证永久纠正措施（PCA）**：如提高限流裕度、加大输出储能、优化补偿；并验证它真能根除，且不引入新问题。
- **D6 实施并确认**：正式改设计/BOM，小批量验证 + 量产监控，确认长期有效。
- **D7 防再发**：把教训固化——**更新设计规范（限流裕度必须叠加瞬态尖峰 + 温漂）、评审 checklist、选型计算模板**，让下一个项目、下一个人不再踩。
- **D8 总结与表彰**：归档整个证据链，团队复盘，让知识沉淀下来。

**证据链**（回链番外01）：从现象波形、复现条件、二分观测点、隔离实验记录，到双向确认的对比波形——整条链都要留证据、可追溯。这既是 8D 的要求，也是一份专业失效分析报告和"我觉得应该是……"之间的根本区别。

## 🔧 工程师速查（附录）

### 六步故障定位法（一句话流程）

**复现**（把偶发逼成常发：加负载/加温/拉偏/加次数/加扰动）→ **二分**（沿供电/信号链半分法，每次砍一半）→ **假设**（鱼骨图 6M 横向铺全 + 5 Why 纵向挖深）→ **隔离**（一次只动一个变量：冷冻剂/热风枪局部激发 + 单因子替换）→ **确认**（能复现 + 能消除，双向可控才算真凶）→ **防再发**（8D 闭环 + 证据链 + 固化规范）。

### 偶发故障：提高复现率的五根杠杆

| 杠杆 | 怎么施压 | 揪什么 |
|---|---|---|
| 加负载 / 瞬态 | 满载、突加突卸负载阶跃 | 电源撑不住、限流、欠冲 |
| 加温度 | 热风枪 / 冷冻剂 / 温箱高低温循环 | 温漂、容值/限流/感值变化、结温 |
| 拉偏供电 | 输入拉到规格下限 | 裕度不足类问题 |
| 加次数 / 时间 | 脚本反复上下电、长时间跑（接番外04 自动化） | 低概率偶发 |
| 加扰动 | 敲击 / 弯板 / 插拔 | 虚焊、微裂、接触不良 |

### 鱼骨图 6M 清单（Ishikawa）

| 类别 | 硬件上常见方向 |
|---|---|
| Man 人 | 装错料 / 贴错 / 漏焊 / 混料 |
| Machine 机 | 器件本身补偿/限流/个体偏弱 |
| Material 料 | 电容 DC bias 衰减、电感近饱和、来料漂 |
| Method 法 | 布局、回流路径、环路/软启动设定、工艺 |
| Measurement 测 | 探头带宽/地环路/触发设错 → 先排除测错 |
| Environment 环 | 温度、输入源内阻、湿度、机械应力 |

### 5 Why 示例（TPS54331 偶发复位）

MCU 复位 → 3.3V 瞬间跌破欠压阈值 → 负载突加时输出欠冲过大 → 峰值电流触发逐周期限流被削顶 → 低温下限流下移+电感近饱和使裕度更小 → 选型只按稳态满载算、没叠加瞬态尖峰+温漂。**根因：限流裕度在动态+低温工况不足。**

### 根因确认铁律

- **能复现 + 能消除（能开能关）** 才算真凶：既能主动制造、又能主动消除，因果才闭环。
- **相关 ≠ 因果**：改了个东西不复现，可能只是把工作点推离临界，根因还在。只信双向可控的结论。
- 一次只动一个变量；改完记录、换回复验。

### 二分定位（半分法）要点

- 沿信号链 / 供电链分段，在中间设观测点或切割点。
- 每次一刀砍两半，判断故障落在哪半段——范围指数级收窄。
- 先分"受害者 vs 真凶"（如：是电源跌了还是 MCU 自己的问题），再往下细分。
- 定位阶段尽量"只观测不换件"，换件留到隔离阶段做单因子验证。

### 8D 八步速查（D0–D8）

| 步 | 做什么 |
|---|---|
| D0 | 应急止血（临时遏制、保护客户/出货） |
| D1 | 组跨职能团队 |
| D2 | 5W2H 量化描述问题 |
| D3 | 临时遏制措施 ICA + 验证 |
| D4 | **找根因并双向验证**（本篇六步的核心） |
| D5 | 选永久纠正措施 PCA + 验证 |
| D6 | 实施 + 小批量/量产确认 |
| D7 | 防再发：固化规范 / checklist / 选型模板 |
| D8 | 归档证据链 + 复盘表彰 |

### 常见翻车

- 上来就换件（换件≠定位，蒙对≠找到）。
- 一次动多个变量，故障消失说不清是谁的功劳。
- 把"测量误差/探头问题"当成真故障（先排除测错）。
- 做到"不复现"就收工，没做"能开能关"的双向确认。
- 只修当前这块板，没做 D7 防再发，同类坑换个项目又出现。
