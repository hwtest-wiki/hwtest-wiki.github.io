# 手测 VS AI 测：同一块板的 I2C 信号质量，手测一下午，AI 搭的工具几分钟出报告

<div class="article-meta"><span class="badge">📘 实战番外</span><span class="badge">⏱ 约 16 分钟</span><span class="badge qr">🔧 含工程师速查</span></div>


一块板上八个 I2C 器件，每个都挂在总线上。你要用示波器一个个量：时钟频率对不对、SCL/SDA 的上升下降时间有没有超、过冲下冲压没压到、数据建立保持时间够不够——一个器件十几项，对着规范一项项判达标。

一个器件测下来十来分钟，八个就是一下午。测完这版，换个批次的板子回来，**再从头来一遍**。

问题不在于每一项难——每一项你都会测。问题在于它**又慢、又容易抄错漏项、还很难留下能复查的记录**。你测到第七个器件的时候，手是麻的，眼是花的，某一项到底是 285ns 还是 385ns，回头想复查都找不到当时那张波形。

其实这一整套动作——配示波器、抓波形、量参数、对规范判达标、把结果记下来——**每一步都是照章办事，没有一步需要你临场拍脑袋**。照章办事的事，正是机器最擅长的。

那能不能让示波器自己把这活儿干了？能。而且现在借着 AI，搭这么一个工具，不再是"得先会写一大堆代码"的大工程。

**读完这篇，你能搞懂：**

- ✅ 一个测试**值不值得自动化**，怎么用三条标准快速判断（别一上来就想把所有东西自动化）；
- ✅ 测试自动化的**通用四层骨架**——控制、采集、判据、报告，每一层干什么；
- ✅ **AI 在每一层怎么帮你加速**，以及有一条线 AI 绝对不能碰；
- ✅ 一个真实的 I2C 信号质量自动化，**从手测到一键出报告**是怎么一步步搭起来的；
- ✅ 搭这类工具的**避坑清单**，以及——**什么时候你还是得老老实实手测**。

## 一、先泼盆冷水：不是所有测试都值得自动化

聊自动化之前，得先说清楚一件事——**别一上来就想把所有测试都自动化**。有些测试自动化了省一整天，有些测试你花三天写脚本，结果一辈子就用两次，纯亏。

判断一个测试值不值得自动化，看三条，**三条同时满足，才值得动手**：

1. **仪器可控**：这测试用到的仪器能被程序控制吗？示波器、电源、万用表、数字负载、协议分析仪，绝大多数带 USB/LAN 口的仪器都支持 SCPI 远程指令，能被程序操控。如果测试全靠人眼看、人手拧，那自动化无从下手。
2. **判据可量化**：达标与否能不能写成一条明确的数值规则？比如"上升时间 ≤ 300ns 就算过"。能写成 `if x <= 300` 的，机器就能判；要是得靠老师傅盯着波形"感觉这个沿有点肉"，机器判不了。
3. **高重复 / 要回归**：这测试是不是要反复做很多遍？多个器件、多个条件、每次换批次都要重测——重复次数越多，自动化摊下来越划算。只测一次的，手测更快。

拿开头那个 I2C 信号质量一比：示波器能用 SCPI 控（✅ 满足一），判据是 I2C 规范里写死的时序阈值（✅ 满足二），八个地址、每次换批次都要重测（✅ 满足三）。三条全中，**这就是典型该自动化的活**。

反过来，有些测试三条占不全，就别硬上：EMC 认证测试要专门的暗室和认证机构，跌落、振动是物理动作，热成像要人对着图判读热点——这些要么机器控不了，要么判据没法量化，自动化投入产出比很差。

## 二、自动化的通用骨架：控制、采集、判据、报告

想清楚"该自动化"之后，怎么搭？不管测什么，一套测试自动化都能拆成**四层**，像流水线一样一层接一层：

1. **控制层**——把仪器管起来。用程序连上示波器，配好通道、量程、触发，让它按你要的方式去抓信号。靠的是 SCPI（一套仪器通用的文本指令，比如发 `:MEASure:FREQuency? CHANnel1` 就是问"1 通道的频率是多少"），Python 里用 `pyvisa` 这个库收发这些指令。
2. **采集层**——把数据取回来。触发采集之后，把波形数据、测量读数从仪器读到电脑里，变成程序能处理的数字。
3. **判据层**——按规矩判 PASS/FAIL。拿采回来的数值，对照规范阈值一项项判：上升时间超没超、电平够不够、时序满不满足。这一层是整套工具的**大脑**，也是最考验你懂不懂测试的地方。
4. **报告层**——把结果留下来。把每一项的实测值、阈值、判定，连同测试条件、时间戳、波形截图，自动汇成一份报告（Excel、HTML 都行），谁来复查都清清楚楚。

这四层不是我硬凑的。我自己搭的那个 I2C 示波器分析工具，代码结构就是照这四层分的：一个模块专管示波器通信（控制+采集），一个模块专管信号质量判定（判据），一个模块专管生成 Excel 报告（报告），再加一个协议解码模块顺带把 I2C 的 START/STOP、地址、数据、ACK 从波形里认出来。**分层分得清楚，整个工具就好搭、好改、好查错。**

![图3：测试自动化的四层架构链路图。从左到右四个方块串成流水线，每个方块上标该层的职责、用到的技术、以及"I2C 工具里对应哪个模块"：① 控制层（pyvisa + SCPI 控示波器：连接、配通道/量程/触发）→ ② 采集层（触发采集、读回波形数据与测量读数）→ ③ 判据层（对照 I2C 规范阈值逐项判 PASS/FAIL，标注"这是工具的大脑"）→ ④ 报告层（汇总实测值+阈值+判定+条件+时间戳+波形截图，出 Excel/HTML）。下方一条虚线框标注"协议解码：从波形认出 START/STOP、地址、R/W、数据、ACK/NACK"，挂在采集与判据之间。整张图强调：不管测什么，都能套这四层](/images/b04/ill03_arch.png)

## 三、AI 在每一层怎么帮你加速——以及那条不能碰的线

前面那四层，放在几年前，得你自己啃仪器编程手册、查 Python 库、慢慢调，没点软件底子搭起来很费劲。现在有了 AI，每一层它都能帮上忙：

- **控制层**：以前查一条 SCPI 指令要翻几百页的编程手册。现在你直接问 AI"帮我用 pyvisa 写一个函数，连上 Keysight 示波器、读回 1 通道的上升时间"，它就给你一段能跑的骨架，你在真机上调通即可。
- **采集层**：示波器读回来的波形常是一串二进制或 CSV，格式说明藏在手册角落。把格式描述丢给 AI，让它帮你写解析代码，省掉大量对着手册数字节的功夫。
- **判据层**：判定逻辑（"把实测值和阈值比一比、超了标红"）这种代码，AI 写得又快又好。
- **报告层**：让 AI 帮你用 `openpyxl` 生成 Excel、或者用 PySide6 搭个小界面，模板类的代码它最拿手。
- **查错**：仪器通信最容易卡在超时、编码、地址格式这些坑上。把报错贴给 AI，它往往一眼看出问题。

但是——有一条线，**AI 绝对不能碰：判据的阈值数值**。

上升时间到底该 ≤300ns 还是 ≤120ns？数据建立时间的下限是多少？**这些数字必须来自 I2C 官方规范（NXP UM10204）或器件的 datasheet，一个字都不能让 AI 编。** AI 是语言模型，你要是问它"I2C 快速模式上升时间上限是多少"，它可能给你一个听起来很像、实际错了的数——而判据错一个数，整套工具就在替你说谎，测出来的"全过"是假的。

所以正确的姿势是：**判据的数值你查规范、亲手填进去；判据的代码骨架让 AI 写。** 数值归你，代码归 AI，各司其职。这一条，是这篇里最重要的一句话。

![图4：AI 加速五环 + 判据红线。中间五个环形依次标注 AI 能帮的五件事：① 控制（生成 SCPI/pyvisa 封装）② 采集（写波形/CSV 解析）③ 判据（写"比阈值判 PASS/FAIL"的逻辑代码）④ 报告（生成 openpyxl/PySide6 模板）⑤ 查错（VISA 超时/编码/地址报错定位）。图正中一条醒目的红色禁止带压在"判据"环上，写着"判据阈值数值：必须查 UM10204 / datasheet，绝不让 AI 编"。底部一句总结：数值归你、代码归 AI——AI 是加速器，不是判据的来源](/images/b04/ill04_ai.png)

## 四、从手测到一键出报告：一个 I2C 信号质量自动化走一遍

讲了骨架，来走一个真实的例子。测一颗公开的温度传感器 TMP102（I2C 地址 0x48），看它挂在总线上时，SCL/SDA 的信号质量达不达标。

**第一步：先把手测流程写下来。** 自动化不是凭空造，而是把你手测的每一步翻译成代码。手测时你做的是：
1. 把示波器 SCL、SDA 两个通道接好，配好量程和触发；
2. 让总线上有 I2C 通信（比如反复读 TMP102 的温度寄存器）；
3. 触发抓一帧波形；
4. 量：SCL 频率、SCL/SDA 上升下降时间、高低电平、过冲、数据建立/保持时间……十几项；
5. 每一项对着 I2C 规范判达标；
6. 把结果记下来。

**第二步：圈出哪些能自动化。** 第 1、3、4、5、6 步全是照章办事，能自动化；第 2 步要 DUT 真的在通信，这个前提得你保证。

**第三步：先用"模拟模式"把流程跑通，别急着接真机。** 这是个特别值钱的习惯——先让工具在**不接示波器**的情况下，用一组假的波形数据把"采集→判据→报告"整条链路走通。这样你能先把判据逻辑、报告格式全调对，等接上真示波器时，只需要调通"控制+采集"这一层，问题范围小得多。我那个工具就专门做了模拟模式，无示波器也能体验完整流程。

**第四步：接真机，一键跑。** 模拟通了，插上示波器，自动检测 VISA 地址、连接，选好 SCL/SDA 通道，点开始。工具自动配示波器、触发采集、量十几项参数、对照 I2C 规范逐项判 PASS/FAIL，还顺手把波形解码成 START/地址/数据/ACK。

**第五步：出报告。** 跑完自动生成一份 Excel：概要（哪个地址、总判定 PASS/FAIL）、信号质量（每项实测值 + 规范阈值 + 判定）、协议解码、波形截图，全都带上。八个地址就出八份，换批次重跑一遍即可——**当初一下午的活，现在几分钟。**

![图5：TMP102 端到端闭环示例。用五个环节的循环流程图展示："手测流程"（左，列出接通道→让总线通信→触发抓波→量十几项→对规范判→记录）→"圈可自动化环节"（标出除"保证 DUT 通信"外全部可自动）→"模拟模式跑通"（不接示波器，用假数据把采集/判据/报告链路调对，标一颗星"最值钱的习惯：先模拟后真机"）→"接真机一键跑"（自动检测/连接/配置/触发/测量/判定/解码）→"出 Excel 报告"（概要/信号质量表/协议解码/波形截图）。整个循环旁标一句对比："一下午 → 几分钟；换批次重跑一遍即可"。全程用公开器件 TMP102@0x48，不含任何产品信息](/images/b04/ill05_walkthrough.png)

## 五、搭这类工具的避坑清单

真动手时，下面这几个坑基本人人都会踩，提前知道能省很多时间：

- **先让一条 SCPI 指令跑通，再组装整个工具。** 别一上来就写几百行。先在命令行里手发一条 `*IDN?`（问仪器"你是谁"），能收到回复，说明通信链路是通的，再往上搭。
- **模拟模式先行。** 前面说过，先用假数据把"采集→判据→报告"调对，再接真机调"控制+采集"。分两段调，比一锅端好排查十倍。
- **VISA 通信三个常见坑**：① 超时——采集慢的操作要把 timeout 调大，不然还没读完就报错；② 编码——有些仪器返回带特殊字符，要指定好编码方式；③ 地址格式——USB 的 VISA 地址一长串，用自动检测别手敲。这些报错贴给 AI 基本能秒解。
- **判据阈值集中管理，别散落在代码各处。** 把所有阈值放一个配置文件或一张表里，来源标清楚（哪个规范、哪一版）。将来规范更新、或要支持不同速率模式，改一处就行，也方便复查。
- **报告一定要留痕。** 时间戳、测试条件、仪器型号、波形截图，全存进去。自动化的一大价值就是**可复查**——半年后有人质疑某个数据，你能立刻翻出当时的原始波形。

![图6：搭测试自动化工具的避坑清单卡。一张清单卡片，五条各配一个图标和一句"为什么"：① 先跑通一条 SCPI（*IDN? 收到回复再往上搭）——验证链路通不通；② 模拟模式先行（假数据调采集/判据/报告，再接真机调控制/采集）——分段排查省十倍时间；③ VISA 三坑：超时调大 timeout / 指定编码 / 地址用自动检测——贴给 AI 秒解；④ 判据阈值集中管理、标清来源版本——规范更新改一处；⑤ 报告留痕：时间戳/条件/型号/波形截图——半年后能复查。卡片底部一句：这些坑几乎人人会踩，提前知道就绕开了](/images/b04/ill06_pitfalls.png)

## 六、什么时候，你还是得老老实实手测

自动化很香，但它有边界。下面这些情况，别想着自动化，老实手测：

- **第一次定位问题时。** 板子出了怪毛病，你得拿探头到处点、来回看波形、边看边想。这种探索性的活靠的是人的判断和经验，没有固定套路可言，自动化插不上手。
- **异常波形要肉眼判读时。** 波形上一个说不清的毛刺、一段奇怪的振铃，机器只会照阈值判过或不过，看不出"这个形状不对劲"。人眼此时不可替代。
- **一次性的验证。** 只测一两次的东西，手测五分钟搞定，写脚本要两小时，纯亏。
- **需要认证机构、或物理操作的测试。** EMC 认证、跌落、振动、老化——机器控不了，或者本就该在专门实验室做。

还有最要紧的一条，回到这篇开头的立场：**不懂手测，你写不出对的自动化。** 判据的阈值、正常波形长什么样、异常大概是什么原因——这些认知全来自你亲手测过、踩过坑。自动化是把你**已经懂了**的东西固化成工具，替你重复；它替代不了"懂"这件事本身。所以顺序永远是：**先学会手测，再考虑自动化。**

![图7：手测不可替代的场景卡。四张场景卡片，各配图标：① 首次定位问题——探头到处点、边看边想，探索性判断无固定套路；② 异常波形肉眼判读——毛刺/振铃"形状不对劲"，机器只会判过/不过；③ 一次性验证——手测 5 分钟 vs 写脚本 2 小时，别自动化；④ 认证/物理操作——EMC、跌落、振动、老化，机器控不了。底部一条醒目横幅："不懂手测，写不出对的自动化——先学会手测，再考虑自动化"](/images/b04/ill07_manual.png)

## 七、一句话总结

**能被规则完整描述、又要反复做的测试，就让仪器按四层骨架（控制→采集→判据→报告）自己去测；AI 帮你把每层的代码搭起来，但判据的每一个数字，都得你查着规范亲手填——数值归你，代码归 AI。**

> 速查层：全公开通用方法与公开器件示例，不涉任何产品信息。网站收录完整；公众号只截「四层骨架」那张；视频不含本节。

### 一、该不该自动化 · 三条标准（全中才动手）
| 标准 | 满足的样子 | 不满足就别上 |
|---|---|---|
| 仪器可控 | 仪器带 USB/LAN + 支持 SCPI 远程控制 | 全靠人眼看、人手拧 |
| 判据可量化 | 达标能写成 `if x <= 阈值` | 靠老师傅"感觉" |
| 高重复 / 回归 | 多器件 / 多条件 / 换批次重测 | 只测一两次 |

### 二、四层骨架
控制（pyvisa + SCPI 控仪器）→ 采集（读波形/读数）→ 判据（对规范阈值判 PASS/FAIL，工具大脑）→ 报告（实测值+阈值+判定+条件+时间戳+波形截图，出 Excel/HTML）。

### 三、pyvisa 连接骨架（起手式）
```python
import pyvisa
rm = pyvisa.ResourceManager()
print(rm.list_resources())          # 列出所有 VISA 设备，找到示波器地址
scope = rm.open_resource("USB0::0x2A8D::...::INSTR")
scope.timeout = 10000               # 采集慢的操作把超时调大（ms）
print(scope.query("*IDN?"))         # 先确认通信通：返回厂商/型号/序列号
```

### 四、常用示波器 SCPI 起手式（Keysight 风格，仅示例）
| 目的 | 指令 |
|---|---|
| 问身份 | `*IDN?` |
| 测频率 | `:MEASure:FREQuency? CHANnel1` |
| 测上升时间 | `:MEASure:RISetime? CHANnel1` |
| 测下降时间 | `:MEASure:FALLtime? CHANnel1` |
| 设垂直量程 | `:CHANnel1:SCALe 1` |
| 设边沿触发 | `:TRIGger:EDGE:SOURce CHANnel1` |
| 单次采集 | `:SINGle` |

> 注：具体指令以你仪器的编程手册为准；查指令、封装成 Python 函数可交给 AI，但要在真机上验证。

### 五、I2C 信号质量关键阈值（出处：NXP《UM10204 I2C-bus specification》Table 10）
| 参数 | Standard 100k | Fast 400k | Fast+ 1M |
|---|---|---|---|
| tr 上升时间 (max) | 1000 ns | 300 ns | 120 ns |
| tf 下降时间 (max) | 300 ns | 300 ns | 120 ns |
| tSU;DAT 数据建立 (min) | 250 ns | 100 ns | 50 ns |
| tHD;DAT 数据保持 (min) | 0 | 0 | 0 |
| tSU;STA 重复起始建立 (min) | 4.7 µs | 600 ns | 260 ns |
| tHD;STA 起始保持 (min) | 4.0 µs | 600 ns | 260 ns |
| tSU;STO 停止建立 (min) | 4.0 µs | 600 ns | 260 ns |
| tBUF 总线空闲 (min) | 4.7 µs | 1.3 µs | 500 ns |
| tSP 尖峰抑制 (max) | — | 0–50 ns | 0–50 ns |
| 电平 | VIH = 0.7×Vdd，VIL = 0.3×Vdd | | |

> ⚠️ 这些数值必须来自规范/datasheet，**不能让 AI 生成**。判据阈值集中放一处、标清来源版本。

### 八、什么时候别自动化
首次定位问题 · 异常波形肉眼判读 · 一次性验证 · 认证/物理操作（EMC、跌落、振动、老化）。**不懂手测，写不出对的自动化——先学会手测，再考虑自动化。**
