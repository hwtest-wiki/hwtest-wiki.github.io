# 同样是 256G 手机，为什么有的秒开大型游戏、有的卡到怀疑人生？——UFS：把 eMMC 的并行老总线换成 MIPI M-PHY 串行，一致性测法为什么和显示家族"分了家"

先说个很多人都纳闷过的现象。

你和同事都买了标称 **256GB** 的手机，跑分软件一测，存储读写速度差了一大截：一个顺序读能到 **每秒三四 GB**，另一个只有 **每秒几百 MB**；装大型游戏、拷一段 4K 视频、系统更新解压，快的那台明显利索，慢的那台转圈转到你怀疑人生。查参数才发现：快的用的是 **UFS 4.0 / 3.1**，慢的用的是 **eMMC**。

再比如做嵌入式的同学会发现，同样一颗闪存，接 **eMMC** 时 PCB 上密密麻麻拉了 **十来根等长线**（数据 + 时钟 + 命令），一到高速就为对齐这些线头疼；换成 **UFS** 之后，就剩 **几对差分线**，PCB 反而清爽了，速度还翻了好几倍。

这两件事背后，是**存储接口从并行到串行的一次架构换代**：从 eMMC 那条"并行总线 + 转发时钟"，换成了 UFS 的"MIPI M-PHY 串行差分 + 嵌入式时钟 + 全双工 + 命令队列"。而测试工程师要保证"这颗 UFS 器件能跑到标称 Gear、跟主控稳定互通"，用的又是一套和显示家族既像又不像的测法——**这，就是这一篇要讲的。**

## 一、先认清主角：UFS / eMMC 是什么、就在你兜里

**UFS（Universal Flash Storage，通用闪存）** 是 JEDEC 定义的高性能嵌入式存储接口，专门装在**空间和功耗都紧张**的设备里：手机、平板、车机、可穿戴、部分工业与消费电子。它对上给操作系统一套 **SCSI 命令**（所以系统眼里 UFS 更像一块小 SSD），对下用 **MIPI M-PHY** 高速串行差分把数据搬进搬出。

它取代的是 **eMMC（embedded MultiMediaCard）**——一种把 NAND 闪存 + 控制器封在一起、走**并行总线**的老接口。eMMC 现在还没死，低端手机、IoT、便宜的嵌入式设备里仍常见，但高端设备早已全面 UFS 化。

UFS 有三大结构特征，正好对应这一篇的三条测试主线：

1. **串行 M-PHY + 嵌入式时钟 → 靠 CDR**：没有独立时钟线，时钟埋在数据里，接收端用时钟恢复（CDR，↔第 32 篇）捞出来——**所以要看眼图**。
2. **全双工**：M-PHY 每个方向有独立的 TX / RX 差分 lane，读和写可以同时进行（eMMC 是半双工，同一时刻只能读或只能写）。
3. **命令队列（SCSI-based）**：主控可以一次抛出多条命令排队执行，不用等一条完成再发下一条——这是 UFS 比 eMMC 快出一个数量级的关键之一。

![图2：认清 UFS 主角。中间一条 UFS 链路连 Host（SoC/主控）↔ Device（UFS 闪存器件）。画清三件事：①M-PHY 物理层——每方向独立 TX/RX 差分 lane（最多 2 lane），高速串行、嵌入式时钟；②UniPro 协议栈跑在 M-PHY 之上做可靠传输；③对上是 SCSI 命令、支持命令队列。旁边标身边场景：手机/平板/车机/可穿戴/部分 SSD。底部点出三大结构特征：①串行 M-PHY 嵌入式时钟→靠 CDR（要看眼图）②全双工（读写同时，eMMC 半双工）③命令队列（多命令排队，eMMC 做不到）。一句话：UFS 的物理层是 MIPI M-PHY，天生站在"看眼图"那岸](/images/45/ill02_whatis.png)

## 二、原理与演进：从 eMMC 并行，到 UFS 的 M-PHY 串行 + 分层协议栈

### 2.1 分层：像网络协议一样一层层叠上去

UFS 不是"一根总线走天下"，而是一套**分层协议栈**，从上到下：

- **应用层（Application）**：UFS 命令集，基于 **SCSI**——系统把它当块存储设备访问。
- **UTP（UFS Transport Protocol，传输协议层）**：把上层命令打包成 UPIU 传输单元。
- **UniPro（Unified Protocol，MIPI 定义的统一协议）**：负责可靠传输、流控、纠错，内部又分 L1.5（PHY 适配）/ L2（数据链路）/ L3（网络）/ L4（传输）几层。
- **M-PHY（MIPI 物理层，L1）**：真正把比特流以高速差分信号发出去的一层。

**关键认知：UFS = SCSI 命令 + UniPro 协议 + M-PHY 物理层。** 其中 UniPro 和 M-PHY 都是 **MIPI 联盟**的东西——这也是为什么我们说"存储和 MIPI 是一门 PHY"。

### 2.2 关键分水岭：并行 eMMC vs 串行 M-PHY

这是本篇最重要的一张对比，也是"从看时序搬到看眼图"的分水岭：

- **eMMC = 并行总线 + 转发时钟**：最多 8 根数据线 + 1 根命令线 + 1 根时钟线并排走，所有线要**严格等长对齐**；接收端**不用 CDR**，直接按那根转发过来的时钟采样。测的是 **setup/hold 时序裕量**，不是眼图。速率一高，并行线间的 skew、串扰就成了瓶颈——这就是并行接口的天花板。
- **UFS = 串行差分 M-PHY + 嵌入式时钟**：几对差分 lane，**时钟埋在数据里**，接收端靠 **CDR** 恢复。测的是**眼图 + 抖动**，和 USB3 / PCIe / DP 完全同路。串行差分天生抗噪、好扩速，这就是 UFS 能一路冲到 HS-G5（每 lane 23.2 Gbps）的底气。

一句话记：**eMMC 在"看时序"那岸，UFS 靠 M-PHY 跨到了"看眼图 / CDR"这岸**——和显示家族的 DP、外设家族的 USB3 站在了同一边。

### 2.3 全双工 + 命令队列 → 为什么快出一个数量级

光有高速串行还不够，UFS 快还靠两件事：**全双工**（读写同时进行，M-PHY 独立 TX/RX lane）+ **命令队列**（多条命令排队、乱序完成，像 PC 上 SSD 的 NCQ）。eMMC 半双工、命令基本串行，所以哪怕拉到 HS400（400 MB/s），也追不上 UFS 的 GB/s 级。

![图3：UFS 的两大骨架。上半——分层协议栈：应用层(SCSI/UFS 命令) → UTP(打成 UPIU) → UniPro(L1.5/L2/L3/L4，MIPI 定义) → M-PHY(L1，MIPI 物理层)，右侧标注"UniPro+M-PHY 都是 MIPI 家的"。下半——并行 vs 串行分水岭：左 eMMC = 8 根并行数据+命令+1 根转发时钟、要等长对齐、接收端不用 CDR、测 setup/hold 时序（"看时序"岸）；右 UFS = 几对差分 lane、嵌入式时钟、靠 CDR、测眼图（"看眼图"岸），并标全双工(独立 TX/RX)+命令队列。底部一句：从 eMMC 到 UFS，就是并行换串行、看时序搬到看眼图](/images/45/ill03_arch.png)

## 三、版本与速率：从 HS-G1 到 HS-G5 的 Gear 阶梯

UFS 的速率用 **M-PHY 的 HS-Gear（高速档）** 表示，每一档又分 **Rate A / Rate B** 两个略不同的频点，全部用 **8b/10b 编码**（20% 开销）。主流版本对应关系（均为 2 lane）：

| 版本 | 最高 Gear | 每 lane 线速率 | 2 lane 总线速率 | 约合原始带宽 |
|---|---|---|---|---|
| UFS 2.0 / 2.1 | HS-G3 | 5.0 / 5.8 Gbps | ≈ 11.6 Gbps | ≈ 1.16 GB/s（有效，8b/10b 后） |
| UFS 3.0 / 3.1 | HS-G4 | 11.6 Gbps | ≈ 23.2 Gbps | 2.9 GB/s 原始 / ≈ 2.3 GB/s 有效 |
| UFS 4.0 | HS-G5 | 23.2 Gbps | ≈ 46.4 Gbps | 5.8 GB/s 原始 / ≈ 4.6 GB/s 有效 |

要点：

- **HS-Gear 逐档翻倍**：G1 1.25/1.45G → G2 2.5/2.9G → G3 5.0/5.8G → G4 11.6G → G5 23.2G（Gbps/lane，Rate A/B）。
- **8b/10b 开销**：每 8 bit 用 10 bit 传，20% 开销——所以"原始线速率 ÷ 8"的 GB/s 数字要再打个八折才是有效带宽（UFS 4.0：46.4G × 0.8 ÷ 8 ≈ 4.6 GB/s）。
- **还有 LS（PWM）低速档**：链路启动、控制、低功耗时用，速率只有几~上百 Mbps，不用 8b/10b——这套"高低速双模"和下一篇 MIPI D-PHY 的 HS/LP 双模是同一个思路。

![图4：UFS 版本与速率阶梯。横向从左到右：UFS2.0/2.1 HS-G3 5.8G/lane（2 lane≈11.6G）→ UFS3.0/3.1 HS-G4 11.6G/lane（≈23.2G，2.9GB/s 原始）→ UFS4.0 HS-G5 23.2G/lane（≈46.4G，5.8GB/s 原始/~4.6GB/s 有效）。每档标 Rate A/B 两频点、8b/10b（20% 开销）虚线框。右端大字标 UFS4.0 = HS-G5 × 2 lane = 46.4Gbps。底部一行小字：另有 LS/PWM 低速档（链路启动/低功耗，几~上百 Mbps，不用 8b/10b）——高低速双模和 MIPI 一个思路](/images/45/ill04_versions.png)

## 四、番外主角：eMMC——正在退场的并行老兵

这一篇的第二主角是 **eMMC**，它是理解 UFS"为什么这么设计"的最好对照物。

- **物理层**：并行总线，最多 **8 bit 数据 + 命令线 + 时钟线**，源同步（转发时钟），接收端不用 CDR。
- **速率天花板**：eMMC 5.1 的 **HS400** 模式约 **400 MB/s**——这已经是并行接口拉到极限的样子，再快 skew / 串扰 / 等长就顶不住了。
- **半双工 + 浅队列**：同一时刻只能读或只能写；eMMC 5.1 才加了简单的命令队列（CQ），但深度和 UFS 的 SCSI 队列没法比。
- **测法完全不同**：eMMC 走的是**并行时序测试**——测各数据线相对时钟的 **setup/hold 裕量、skew**，用示波器多通道量时序，**没有"眼图 + CDR + 参考均衡"这套**。这正好反衬出 UFS 换成 M-PHY 串行后，测法整个搬到了眼图这一岸。

一句话：**eMMC 是"并行 + 看时序"的老世界，UFS 是"串行 M-PHY + 看眼图"的新世界**；低端和 IoT 里 eMMC 还在服役，但趋势是被 UFS 全面取代。

![图5：eMMC——并行老兵（对照 UFS）。左：画 eMMC 并行总线 = 8 根数据 + 命令线 + 1 根转发时钟并排，标"要等长对齐、接收端不用 CDR"。中：速率天花板 eMMC5.1 HS400 ≈ 400MB/s（并行到顶），半双工、浅命令队列(CQ)。右：测法对比框——eMMC 测 setup/hold 时序裕量 + skew（多通道示波器量时序，无眼图）vs UFS 测眼图 + 抖动 + CDR。底部一句：eMMC=并行看时序老世界，UFS=串行 M-PHY 看眼图新世界，低端/IoT 仍在用但正被取代](/images/45/ill05_emmc.png)

## 五、测什么·三类：Tx 眼图、Rx 压力测试、封装 / 通道

UFS（HS-Gear）的一致性测试，和 USB3 / DP 一样分三大阵营——**全程都在"CDR + 眼图"的世界里**：

1. **Tx / 发送端眼图**：示波器在规定测试点抓 M-PHY 差分 lane 的眼图，扣 **M-PHY CTS 的眼图模板** + 抖动分离（RJ / DJ / TJ，↔第 31 篇），并验各档 **去加重 / 摆幅**。Gear 越高（G4/G5）模板越严。
2. **Rx / 接收端压力测试**：用 BERT / 码型发生器造**校准过的压力眼**（把损耗 + 抖动加到规范最坏值），测器件的**抖动容限 / 接收灵敏度 / CDR 跟踪能力**（↔第 41 篇）。这一步验的是"最坏信号来了它还认不认得"。
3. **封装 / PCB 通道**：UFS 器件焊在板上，没有可插拔线缆，但**封装球脚→PCB 走线→器件**这段通道的插损 / 回损 / skew / 串扰同样要测（VNA / 示波器），高 Gear 下这段通道就是裕量杀手。

![图6：UFS 一致性测试三阵营（全程 M-PHY CTS + 眼图世界）。左 Tx/发送端：示波器在测试点抓 M-PHY 差分 lane 眼图→扣 M-PHY CTS 眼图模板 + 抖动分离(RJ/DJ/TJ) + 去加重/摆幅各档；标 G3 模板 vs G4/G5 模板(更严+参考均衡)。中 Rx/接收端：BERT/码型发生器造校准压力眼(loss+jitter 加满)→测抖动容限/接收灵敏度/CDR 跟踪(↔41)。右 封装/通道：VNA/示波器测封装球脚→PCB 走线→器件的插损/回损/skew/串扰(高 Gear 重灾区)。底部标一句：UFS 没有可插拔线缆，通道就是封装+PCB 那段](/images/45/ill06_test3.png)

## 六、怎么测：进测试模式 + link startup 训到目标 Gear + 参考均衡

UFS 的测试流程，几乎照搬 USB3 / DP 那套（"学会一个通一片"）：

- **进测试模式**：通过 UniPro / M-PHY 的管理接口（**RMMI / RMC，M-PHY 管理与配置接口**）把器件切进测试态，让它固定输出 **PRBS 测试码型**、锁定目标 Gear（G4 / G5）。
- **link startup（链路启动）**：主控和器件先从 **LS（PWM）低速档**握手（UniPro 的 **PACP** 流程），再逐档往 HS 上试、协商 lane 数 / 速率 / 均衡参数，训到稳定档；训不上就回落到低 Gear（表现为掉速）。
- **测量**：示波器在测试点抓眼图套 M-PHY CTS 模板；BERT 灌压力眼测容限。**G4 / G5 必须挂参考均衡（CTLE / DFE，↔第 34 篇）**，把高频损耗补平、把闭着的眼睁开再判读——和 USB3 / PCIe / DP 同一个道理。

![图7：UFS 测试台面搭建。左区块"进测试模式"：上位机经 RMMI/RMC 管理接口→被测器件固定输出 PRBS 测试码型、锁定目标 Gear(G4/G5)。中区块"link startup"：Host↔Device 先 LS/PWM 低速握手(UniPro PACP)→逐档往 HS 上试→协商 lane 数/速率/均衡→训到稳定档，训不上则回落(掉速)。右区块"测量"：测试点接示波器抓眼图套 M-PHY CTS 模板；接 BERT 灌压力眼测容限。底部强调框：G4/G5 必挂参考均衡 CTLE/DFE(↔34) 才能把眼睁开判读——和 USB3/PCIe/DP 同路](/images/45/ill07_bench.png)

## 七、⚙️ 关键仪器设置：把模块三的 SI 概念落到旋钮上

UFS 的仪器设置和 USB 3.x / DP UHBR 那套几乎一模一样：

- **去嵌（de-embed）↔第 33 篇**：把探测夹具 / 通道的损耗从测量结果里数学扣掉，还原器件球脚处的真实信号。
- **参考均衡 CTLE / DFE↔第 34 篇**：G4 / G5 必用，把高频损耗补平——**判据是"参考均衡之后"的眼图**，不是裸眼图。
- **时钟恢复 golden PLL↔第 32 篇**：M-PHY 是嵌入式时钟，示波器要用规范规定的黄金 PLL 环路参数恢复时钟，才能正确测抖动。
- **带宽按规范档设↔第 29 / 速查 02**：按波特率约 0.7～1 倍选示波器带宽（G5 约 23G，用 30～40G 级示波器足矣，别过冲）。
- **抖动分离 + eye mask↔第 31 / 32 篇**：把 TJ 拆成 RJ / DJ，再套 M-PHY 眼图模板判裕量。

> 一句话：**UFS 的仪器设置 = USB 3.x / DP 那套嵌入式时钟测法。** 你在第 41 / 44 篇学的东西，这里几乎能直接复用；eMMC 那套并行时序测法（量 setup/hold）在 UFS 上完全用不到。

## 八、判据从哪来：JEDEC + MIPI 两本 CTS，不是拍脑袋

和第 38 篇讲的一样，UFS 的判据来自标准的 **CTS（Compliance Test Specification，一致性测试规范）**，而且是**两家、两本**：

- **M-PHY CTS（MIPI）**：管电气层——眼图模板、抖动限值（RJ / DJ / TJ）、去加重 / 摆幅、接收均衡要求。眼图模板 / 抖动预算从信道损耗预算反推而来（↔第 38 篇）。
- **UFS 一致性测试（JEDEC）**：管协议 + 电气整体符合性。JEDEC 维护 UFS 标准（如 JESD220 器件、JESD223 主控 UFSHCI）及对应的一致性测试规范，验 link startup、UniPro 行为、命令队列、UPIU 等协议项。

所以 UFS 的判据体系是**分家的**：**电气 / PHY 归 MIPI M-PHY CTS，协议 / 命令归 JEDEC UFS 一致性**——两大本分别由不同测试系统跑。这一点和 DP 的"PHY CTS + Link Layer CTS 两本"结构上完全类似。

## 九、常见翻车与调试：掉速 / 训不上高 Gear / 兼容不上 / 偶发读写错，怎么分层揪

UFS 翻车，**先分层、别一上来就赖信号**（顺序和 USB / DP 一样）：

1. **先看协商 / 训练层（link startup / Gear）**：器件被识别没？link startup 训到第几 Gear？——**"读写掉速""跑分只有一半"多半是没训到目标 Gear、回落了**，先查这里，跟眼图无关。
2. **再看电气层（眼图 / skew / 均衡 / 通道）**：训到高 Gear 但偶发误码、花屏（车机屏）、掉盘——量 M-PHY 差分眼图、封装 / PCB 通道 skew、参考均衡后眼还睁不睁、球脚阻抗是否连续。
3. **协议 / 命令层（UniPro / 命令队列）**：链路稳但吞吐上不去、并发一多就出错——查 UniPro 流控、命令队列深度、UPIU 处理。
4. **供电 / 完整性层（PI，↔模块三）**：高负载读写时纹波 / 掉压导致偶发错误——存储大电流突发时 PI 尤其关键。
5. **eMMC 兼容 / 老器件**：若是 eMMC，问题多在并行 setup/hold 时序、等长走线、HS400 的数据选通（DS）——那是另一套并行世界的排查法。

**现象速查**：跑分只有标称一半 = 没训到目标 Gear / 回落；训到高 Gear 仍偶发误码 = 眼图 / 通道 skew / 均衡；吞吐上不去但链路稳 = UniPro 流控 / 命令队列；高负载才偶发掉盘 = 供电 PI；eMMC 高速不稳 = 并行 setup/hold / 等长 / DS 选通。

## 一句话总结

**一句话：从 eMMC 到 UFS，是存储接口从「并行 + 转发时钟 + 看 setup/hold 时序」换代到「MIPI M-PHY 串行差分 + 嵌入式时钟 + CDR + 看眼图」——一致性测试也随之整个搬到 USB3 / PCIe / DP 那一岸。学会了 USB3 / DP，UFS 你已经会了一大半，MIPI 也通了半只脚。**

- **换代**：并行 + 转发时钟 + setup/hold 时序（eMMC）→ M-PHY 串行差分 + 嵌入式时钟 + CDR + 看眼图（UFS）。
- **怎么测**：抓 M-PHY 眼图、做抖动分离、G4/G5 挂参考均衡。
- **判据两本**：MIPI M-PHY CTS（电气）+ JEDEC UFS（协议）。
- **承上启下**：UFS 的物理层 M-PHY，正是下一篇 MIPI 的同门兄弟。
