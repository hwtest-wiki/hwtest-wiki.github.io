# 同样一根 HDMI 线，为什么有的能 4K120 / 8K、有的一插就花屏黑屏？——HDMI：从 TMDS 到 FRL，测法怎么跟着传输机制换了一套

先说个几乎人人都遇到过的现象。

你换了台新电视，标称 **4K 120Hz / 8K**，兴冲冲接上游戏机或电脑。有时候一切完美；有时候一插——**黑屏、雪花、闪一下就没信号，或者分辨率被硬压到 4K30、颜色发绿发暗**。换根"看起来一模一样"的 HDMI 线，居然又好了。

问题来了：**接口、插头、分辨率标称都一样，为什么行不行差这么多？**

答案藏在一个大多数人没意识到的事实里：**HDMI 这个口，内部悄悄换过一次"发动机"。** 老设备用 TMDS，HDMI 2.1 的高分辨率靠 FRL，两者的物理传输方式**根本不是一回事**。而测试工程师要保证"这个 HDMI 口 / 这根线跟全世界的电视都能通"，就得分别按这两套机制去测——**这，就是这一篇要讲的。**

## 一、先认清主角：HDMI 是什么、身边到处都是它

**HDMI（High-Definition Multimedia Interface，高清多媒体接口）** 是把**视频 + 音频 + 控制信息**打包，用一根线在一个"源"和一个"显示端"之间单向传输的接口。用协议里的话说，一端叫 **Source（源，如游戏机、电脑、机顶盒）**，另一端叫 **Sink（宿 / 显示端，如电视、显示器）**。

它有多普及不用多说：电视、显示器、游戏机、机顶盒、投影仪、笔记本的投屏口、相机的取景外接……**客厅和办公桌上最常见的那个梯形口，基本就是它。** 也正因为无处不在、面对全世界成千上万种电视和线材，它才必须做**一致性测试**——保证"跟谁都能通"。

和前面讲的 USB 相比，HDMI 有两个鲜明特点，直接决定了它怎么测：

- **它是"显示专用、基本单向"的**：Source 一个劲儿往 Sink 送像素，没有 USB 那种对等的双向数据。所以测试天然分成 **Source（发送端）测试**和 **Sink（接收端）测试**两大阵营。
- **它自带一条边带小道 DDC / SCDC**：两端靠这条低速通道互相"通报家底"——你支持什么分辨率、支持不支持 FRL、要不要切模式。这条小道虽然不跑视频，却是**能不能协商到高规格**的关键（很多"掉分辨率"就死在这里）。

## 二、原理与演进：老发动机 TMDS，新发动机 FRL

HDMI 的物理层历史，可以粗暴地分成"FRL 之前"和"FRL 之后"。

### 2.1 老发动机：TMDS（HDMI 1.x / 2.0）

**TMDS（Transition Minimized Differential Signaling，最小化跳变差分信号）** 是 HDMI 从出生就用的方式，从 DVI 继承而来。它长这样：

- **3 条数据通道** + **1 条独立的时钟通道**。三条数据分别扛 R / G / B（或对应的分量），时钟通道**直接把像素时钟发过去**。
- 每条数据通道把 8-bit 像素数据编成 **8b/10b** 的 10-bit 符号（做 DC 平衡 + 控制字符），所以**数据通道的位速率 ≈ 10 × 像素时钟**。
- 时钟是"转发"过去的（forwarded clock / 源同步）：**接收端不用费劲从数据里恢复时钟，直接拿那条时钟通道去采样数据。**

速率上：**HDMI 1.4 总带宽约 10.2 Gbps（每通道 ~3.4 Gbps）；HDMI 2.0 提到 18 Gbps（每通道 6 Gbps）**，编码都是 8b/10b。到 HDMI 2.0，TMDS 基本摸到了天花板——再往上，靠"转发一根时钟 + 8b/10b"这套已经吃力了。

> ⚠️ 这里有个**最容易被误解**的点：TMDS 转发了时钟，是不是就不用看眼图了？**恰恰要看。** 时钟转发只是省了接收端恢复时钟的力气；可三条数据通道跑在 10× 像素时钟的高速上，一样会被抖动、码间串扰、线损搞坏。所以 **HDMI Source 一致性测试从 TMDS 时代起就是"数据通道的眼图 + 眼图模板"**——转发时钟解决的是"何时采样"，眼图解决的是"采样点上信号还睁不睁得开"。这两件事是分开的。

### 2.2 新发动机：FRL（HDMI 2.1）

HDMI 2.1 要上 **4K120 / 8K60** 这种量级，TMDS 那套撑不住了，于是引入了全新的 **FRL（Fixed Rate Link，固定速率链路）**：

- **把那条独立时钟通道也改成数据 lane**——从"3 数据 + 1 时钟"变成**最多 4 条数据 lane**，不再单独发时钟。
- **时钟埋进数据里，接收端靠 CDR（时钟数据恢复）自己捞出来**——这一步，HDMI 就跨进了 USB 3.x / PCIe / DisplayPort 那个"嵌入式时钟"的世界。
- 编码换成更省的 **16b/18b**（效率 ~88.9%，比 8b/10b 的 80% 高）。
- 传输前要先做 **link training（链路训练）**：Source 和 Sink 从低到高试各个 FRL 速率档，看在这根线、这块板上最高能稳到哪一档；训练结果通过 **SCDC** 边带通报。训不上去或线材不够，就**回落**到更低档、甚至退回 TMDS。

FRL 分 6 档，是一张要记住的表：

- **FRL1 = 3 lane × 3G = 9 Gbps**
- **FRL2 = 3 lane × 6G = 18 Gbps**
- **FRL3 = 4 lane × 6G = 24 Gbps**
- **FRL4 = 4 lane × 8G = 32 Gbps**
- **FRL5 = 4 lane × 10G = 40 Gbps**
- **FRL6 = 4 lane × 12G = 48 Gbps**（HDMI 2.1 顶格）

## 三、正确使用与设计要点：线材等级、走线、那条边带小道

在讲测试之前，先说清"用对"，因为一半以上的 HDMI 翻车根本不在芯片，在**线和边带**：

- **线材等级要对上速率**：跑 18G 需要 **Premium High Speed HDMI Cable**，跑 48G FRL 需要 **Ultra High Speed HDMI Cable**（带认证标）。一根标称"支持 8K"却没过认证的便宜线，link training 时上不了 FRL6，直接掉档——这就是"换根线又好了"的物理原因。
- **PCB 走线与连接器**：4 对差分（FRL 下全是数据 lane）要严格等长、控阻抗（差分 100Ω）、少过孔、控 lane 间 skew；连接器和 ESD 器件的寄生会吃掉高频，48G 下尤其敏感。
- **别忽视边带**：**DDC**（读 EDID，知道对端支持啥）和 **SCDC**（HDMI 2.0 引入、FRL 用它报状态 / 切模式）是低速 I²C 类信号，但**它俩一旦不通，能力协商就崩**——现象往往不是花屏，而是"根本协商不到高分辨率"或"忽好忽坏"。
- **HDCP（内容保护）握手**：这不是信号质量问题，但握手失败会直接黑屏。排查"黑屏"时要能把 HDCP 和信号完整性分开看。

## 四、测什么·三类：Source 眼图、Sink 压力测试、线材通道

HDMI 一致性测试（**CTS，Compliance Test Specification**；HDMI 2.1 及 FRL 由 **HDMI Forum** 维护，1.x/2.0 由 HDMI LA 维护）落到电气层，就是三类——和第 38 篇的 **Tx / Rx / Channel** 三分法一一对应：

**① Source（发送端 / Tx）——测"发得干不干净"，看眼图。**
- **TMDS 模式**：在规定测试点抓三条数据通道的波形，叠成眼图，扣 **TMDS 眼图模板**；同时测抖动、上升 / 下降时间、电平。
- **FRL 模式**：让 Source 进 FRL 测试模式、发 FRL 测试码型，抓每条 lane 的眼图扣 **FRL 眼图模板**——这套和 USB 3.x / PCIe 的发送端眼图测试**思路完全一致**（因为 FRL 就是嵌入式时钟那一路）。

**② Sink（接收端 / Rx）——测"脏信号还认不认"，做压力测试。**
- 用码型发生器 / BERT **造一只"最坏压力眼"**（叠加规定的抖动、幅度损耗、ISI），灌给 Sink，看它在这种恶劣条件下**还能不能正确接收、link 稳不稳**——即**抖动容限 / 接收灵敏度**。
- 还要验 Sink 的时钟恢复 / PLL 跟踪能力（FRL 下尤为关键）。这一步和第 41 篇 USB 3.x 的 **Rx jitter tolerance** 是同一套哲学。

**③ Cable / Channel（线材 / 通道）——测"这根线扛不扛得住"。**
- 衰减、lane 间 skew、阻抗 / 回损、串扰。**Ultra High Speed 线的认证**核心就是这块——保证 48G 下信号还能过。

![图6：HDMI 测什么三类（对齐 38 篇 Tx/Rx/Channel）。①Source/Tx：进测试模式发码型→示波器抓数据通道眼图，TMDS 扣 TMDS 眼图模板、FRL 扣 FRL 眼图模板（+抖动/上升沿/电平）。②Sink/Rx：BERT 造最坏"压力眼"（抖动+损耗+ISI）灌进去，测抖动容限/接收灵敏度/PLL 跟踪——与 USB3.x Rx 同哲学。③Cable/Channel：衰减、lane 间 skew、阻抗/回损、串扰——Ultra High Speed 线认证的核心。底部：TMDS 时看老眼图、FRL 时看新眼图，Source 端两套模板](/images/43/ill06_test3.png)

## 五、怎么测：夹具 + 进测试模式 + link training 到目标档 + 示波器 / BERT

台面搭起来大概是这样（沿用第 38 篇的"测试点 + 夹具 + 一致性软件"套路）：

1. **接测试夹具（fixture / TPA）**：把 DUT 的 HDMI 口通过标准测试夹具引出到示波器 / BERT，测试点是规范规定的（连接器处 / 夹具参考面）。
2. **让 DUT 进 compliance test mode、发指定测试码型**：Source 测试要它输出规定图案；FRL 测试还要**先把链路 link training 到你要测的那一档**（比如要测 FRL6，就得先训到 48G）。
3. **示波器 + 一致性软件自动判**：加载对应的 **TMDS / FRL 眼图模板**，软件叠眼图、扣模板、算抖动、出 pass/fail。
4. **Sink 测试上码型发生器 / BERT**：产生带规定损伤的压力信号，扫抖动容限。

![图7：测 HDMI 的台面。中间 DUT（HDMI 源或宿）经测试夹具 TPA 引出。①Source 测：DUT 进 test mode 发码型→（FRL 先 link training 到目标档 FRLx）→示波器 + 一致性软件加载 TMDS/FRL 眼图模板，自动叠眼、扣模板、算抖动、出 pass/fail。②Sink 测：码型发生器/BERT 造压力眼灌入 DUT，扫抖动容限。旁边：边带 DDC/SCDC 用来切模式、报状态。底部：测试点在规范规定的夹具参考面（↔38 测试点/夹具）](/images/43/ill07_bench.png)

## 六、⚙️ 关键仪器设置：把模块三的 SI 概念落到旋钮上

HDMI 的一致性软件里那些选项，全是前面模块三讲过的 SI 概念，这里只做映射、不复述原理：

- **去嵌 De-embed（↔33）**：把测试夹具 / 线缆的影响从测量里扣掉，还原 DUT 真实输出。
- **嵌入 Embed / 参考线缆模型**：有些项要在 DUT 输出上"叠加"一段标准信道 / 最坏线缆，模拟真实到达接收端的样子。
- **参考接收器均衡 CTLE / DFE（↔34）**：**FRL 测试**里，眼图往往要经过规范规定的参考均衡器再判——因为真实 Sink 就带均衡。这是 FRL 比 TMDS 明显"重"的地方。
- **时钟恢复 golden PLL（↔32）**：FRL 是嵌入式时钟，眼图的时间基准来自按规范设定的时钟恢复环路（TMDS 则可用转发的那条时钟做参考）。
- **带宽 ≥ 关注谐波 / 按规范档（↔29 / 速查02）**：示波器带宽按规范要求选；注意**别把"上升时间 5 倍谐波"那条时域规则，硬套到串行眼图的速率上**（详见速查 02——40G 级信号并不需要 200G 示波器）。
- **抖动分离 + eye mask（↔31 / 32）**：分离随机 / 确定抖动、加载眼图模板判 pass/fail。

## 七、判据从哪来：眼图模板 / 抖动预算不是拍脑袋

和第 38 篇一脉相承：HDMI 那些**眼图模板的开口大小、抖动限值、去加重档位**，不是随手画的，而是从**信道损耗模型 + 抖动预算**反推出来的——规范组织假定了最坏的线缆 / 板子损耗、最坏的接收端能力，倒推出"发送端至少要睁多大的眼、抖动不能超多少"，才能保证任意合规的 Source + Cable + Sink 组合都能通。所以你眼图擦到模板边缘 fail，本质是"留给线缆和接收端的余量不够了"。这套推导逻辑，第 38 篇讲过，这里直接引用。

FRL 相比 TMDS，判据里多了均衡后判定、link training 结果等——因为它是嵌入式时钟链路，天然继承了 USB3/PCIe 那套更复杂的接收机模型。

## 八、常见翻车与调试：黑屏 / 花屏 / 掉分辨率 / 训练失败，怎么分层揪

HDMI 翻车最忌讳"一上来就怀疑信号质量"。正确姿势是**分层**，和第 42 篇"先看协议层、再看电气层"一个思路：

- **协商层（先看这里）**：读 EDID 对不对、DDC/SCDC 通不通、HDCP 握手成没成、FRL link training 训到了第几档。**很多"黑屏 / 掉到 4K30 / 忽好忽坏"死在这一层，跟信号眼图无关。**
- **链路 / 电气层**：如果协商 OK 却花屏 / 雪花 / 闪断，再去量眼图、看是不是线材不够级（掉不到目标 FRL 档）、走线 skew、去加重不够、接收端均衡吃不下。
- **内容保护层**：纯黑屏但灯亮、能识别设备，优先怀疑 **HDCP 握手失败**，别跟信号完整性混。

对号入座几个高频现象：

| 现象 | 最可能的层 | 常见根因 |
|---|---|---|
| 一插就黑屏、灯亮能识别 | 内容保护 / 协商 | HDCP 握手失败 / EDID 读失败 |
| 掉到 4K30、颜色发暗 | 协商 | FRL 训不上（线材不够级）→回落 TMDS/低档 |
| 花屏、雪花、间歇闪断 | 电气 | 眼图擦模板 / 线损大 / skew / 均衡不足 |
| 忽好忽坏、插拔才好 | 边带 / 协商 | DDC/SCDC 不稳、EDID 缓存脏 |
| 高刷开不了、只有低刷 | 协商 | Sink 不支持该 FRL 档 / EDID 没报该模式 |

## 九、一句话总结

**HDMI 一个接口，把"转发时钟"和"嵌入式时钟"两套架构走了个来回：老的 TMDS（3 数据 + 1 时钟、8b/10b、≤18G）转发时钟但数据通道照样扣眼图模板；HDMI 2.1 的 FRL（4 lane、埋时钟靠 CDR、16b/18b、最高 48G）跨进了 USB3/PCIe 那个嵌入式时钟世界，眼图前要过参考均衡、上码型还得先 link training。测什么依旧是 Tx 眼图 / Rx 压力 / Cable 通道三类；调试永远先分层——协商训练在哪档、电气眼图睁没睁开、HDCP 握没握上，别混为一谈。**

## 🔧 工程师速查（附录）

| 项 | 要点 |
|---|---|
| HDMI 是什么 | 视频 + 音频 + 控制打包、单向 Source→Sink 的显示接口；另带低速边带 DDC（读 EDID）/ SCDC（报状态、切模式） |
| 两套传输机制 | **TMDS**（HDMI 1.x/2.0）：3 数据通道 + 1 独立时钟、8b/10b、转发时钟；**FRL**（HDMI 2.1）：4 lane 全数据、嵌入式时钟靠 CDR、16b/18b |
| 时钟架构（↔速查02） | TMDS=转发时钟（省接收端 CDR，但数据通道仍高速串行、仍扣眼图）；FRL=嵌入式时钟 + CDR（跨到 USB3/PCIe/DP 那一类） |
| 版本速率 | HDMI 1.4≈10.2 Gbps（3.4G/ch）；HDMI 2.0=18 Gbps（6G/ch）；HDMI 2.1 FRL 最高 48 Gbps |
| FRL 速率阶梯 | FRL1 3lane×3G=9G｜FRL2 3×6G=18G｜FRL3 4×6G=24G｜FRL4 4×8G=32G｜FRL5 4×10G=40G｜FRL6 4×12G=48G |
| link training | FRL 传输前 Source↔Sink 从低到高试速率定档，结果走 SCDC；训不上→回落低档甚至退回 TMDS |
| 线材等级 | 18G→Premium High Speed；48G FRL→Ultra High Speed（须带认证标）。"换根线又好了"多半是线材不够级 |
| 测什么·三类（↔38） | ①Source/Tx：数据通道眼图，TMDS 扣 TMDS 眼图模板、FRL 扣 FRL 眼图模板（+抖动/上升沿/电平）②Sink/Rx：BERT 造压力眼测抖动容限/接收灵敏度/PLL 跟踪（↔41 Rx jitter tolerance）③Cable：衰减/skew/回损/串扰（Ultra High Speed 认证核心） |
| 关键仪器设置 | 去嵌↔33 · 嵌入参考线缆 · 参考均衡 CTLE/DFE↔34（FRL 必用）· 时钟恢复 golden PLL↔32 · 带宽按规范档↔29 · 抖动分离 + eye mask↔31/32 |
| 带宽误区（↔速查02） | 眼图带宽≈0.7×~1×波特率，别把"上升时间 5 倍谐波"套到串行速率上——48G 级不需要 200G 示波器 |
| 判据从哪来 | 眼图模板 / 抖动限值从信道损耗 + 抖动预算反推（↔38），不是拍脑袋；FRL 判据含参考均衡后判定 + 训练结果 |
| 调试分层 | 先协商层（EDID/DDC/SCDC/HDCP/FRL 训到第几档）→ 再电气层（眼图/skew/均衡/线材级别）→ 分清内容保护层（HDCP 握手）；别一上来就赖信号 |
| 现象速查 | 黑屏灯亮=HDCP/EDID；掉 4K30 颜色暗=FRL 训不上回落；花屏雪花闪断=眼图/线损/skew；忽好忽坏=DDC/SCDC 不稳；开不了高刷=Sink 不支持该 FRL 档/EDID 没报 |
