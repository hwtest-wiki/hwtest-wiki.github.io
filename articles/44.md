# 同样一台高刷显示器，为什么厂商都让你"用 DP 线、别用 HDMI"？——DisplayPort：天生就是嵌入式时钟 + 打包传输，一致性测法和 HDMI 差在哪

先说个很多人都纳闷过的现象。

你买了台高刷电竞显示器，标称 **2K 165Hz** 或 **4K 144Hz**。说明书、客服、甚至显示器背后的贴纸都在提醒你：**"高刷请用 DP 线，HDMI 口可能只能到 60Hz / 120Hz"**。你半信半疑地插了 HDMI，果然——刷新率选项里最高就卡在 60Hz，怎么都上不去；换上附带的那根 DP 线，165Hz 一下就出来了。

再比如，办公室里有人用**一根线从显示器 A 又接出一根到显示器 B**，两台屏都正常显示不同画面——这在 HDMI 上是做不到的。还有你手上这台笔记本，合盖不断电、静态桌面时特别省电、屏幕却一直亮着……

这些看似不相干的现象，背后其实是**同一个接口**在撑：**DisplayPort**。它和 HDMI 长得像、都能投屏，但**骨子里是另一套架构**。而测试工程师要保证"这个 DP 口 / 这块笔记本屏，跟全世界的显示设备都能通、都能跑到标称的高规格"，用的又是和 HDMI 部分重叠、部分不同的一套测法——**这，就是这一篇要讲的。**

## 一、先认清主角：DisplayPort 是什么、身边到处都是它

**DisplayPort（DP）** 和 HDMI 一样，是把**视频 + 音频 + 控制信息**打包、在一个"源"和一个"显示端"之间传输的显示接口，由 **VESA（视频电子标准协会）** 制定。术语上同样分 **Source（源，如显卡 / 笔记本）** 和 **Sink（宿 / 显示端，如显示器 / 内屏）**。

它到底有多常见？其实比你以为的还多：

- **台式机显卡 + 电竞 / 专业显示器**：高刷、高分辨率、多屏，基本走 DP。
- **笔记本外接屏**：无论是 Mini DP 口，还是现在最普遍的 **USB-C 口投屏（DP Alt Mode）**，底层跑的都是 DisplayPort（这条线↔第 42 篇 USB-C）。
- **笔记本 / 平板 / 手机自己的那块内屏**：几乎全是 **eDP（Embedded DisplayPort，嵌入式 DP）** 驱动的——你天天盯着看，却从没见过那根线，因为它直接焊在主板到屏之间。

和刚讲完的 HDMI 相比，DP 有三个鲜明特点，直接决定了它怎么测：

- **它天生就是"嵌入式时钟"的**：DP 从第一版起就**没有独立时钟线**，接收端一律靠 **CDR（时钟数据恢复）** 从数据里捞时钟。这一点 HDMI 到 2.1 的 FRL 才做到，DP 一出生就是。
- **它是"打包（packet）传输"的**：DP 把像素、音频、控制切成一个个微包（micro-packet）在链路上跑，不是 HDMI TMDS 那种"逐像素连续流"。正因为打包，它才能玩 **MST 菊花链**（一根线带多屏）和 **DSC 压缩**。
- **它自带一条能读能写的边带 AUX**：两端靠这条低速**双向**通道读 EDID、读写寄存器（DPCD）、做 link training、协商 HDCP——很多"掉刷新率 / 认不出屏"就死在 AUX 上。

## 二、原理与演进：从出生就是"嵌入式时钟 + 打包"

### 2.1 三件套：Main Link + AUX + HPD

一条 DP 链路，物理上就三部分：

- **Main Link（主链路）**：**1 / 2 / 4 条 lane**（每条一对高速差分），**单向** Source→Sink，扛所有像素 + 音频 + 控制。lane 越多、每 lane 速率越高，能带的分辨率 / 刷新率就越高。
- **AUX Channel（辅助通道）**：一条**低速、双向（半双工）**的小道，负责读 **EDID**（显示器报家底）、读写 **DPCD**（DP 配置状态寄存器）、跑 **link training**、协商 **HDCP** 内容保护。它跑不了视频，却是"能不能协商到高规格"的命门。
- **HPD（Hot Plug Detect，热插拔检测）**：一根信号线，告诉 Source"屏插上 / 拔了 / 有事要报"。

### 2.2 关键分水岭：一直是嵌入式时钟 + CDR（对比 HDMI TMDS）

这是 DP 和 HDMI 最本质的区别，也是它测法路数的根源：

- **HDMI 老的 TMDS**：单独拉一根时钟线转发过去，接收端拿这根时钟直接采样（源同步）。到 HDMI 2.1 的 FRL 才改成嵌入式时钟。
- **DisplayPort：从 DP 1.1 第一版起，就没有独立时钟线。** 时钟是**埋在数据里**的，接收端全靠 **CDR** 从数据流的跳变里把时钟"捞"出来，再拿它采样。

一句话：**DP 一出生就站在"嵌入式时钟 + CDR"这一岸**，也就是 USB 3.x / PCIe 那一路。这直接决定了——**DP 的接收端一定有 CDR、有均衡，一致性测试一定看眼图、要做抖动分离、要用参考均衡还原**，从 HBR 时代就是如此，跟速率高低无关。

> 💡 回到速查 02 的地图：HDMI 是**在分水岭上走了个来回**（TMDS 在"看时序"这侧、FRL 跨到"看眼图"那侧）；**DP 则从头到尾都在"嵌入式时钟 + CDR + 看眼图"那一侧**。所以你会发现，DP 的测法几乎就是 HDMI FRL 那一半的"完整版 + 从头贯彻"。学会一个，另一个就通了大半。

### 2.3 打包传输 → 菊花链 MST 与压缩 DSC

因为 DP 是**打包（micro-packet）**传输而不是连续像素流，它天生能干两件 HDMI 早期干不了的事：

- **MST（Multi-Stream Transport，多流传输）**：一条链路上可以塞**多路独立的视频流**，配合 Hub 或支持菊花链的显示器，**一根线串接多台屏**、各显示各的画面。办公室"一根线带俩屏"就是它。
- **DSC（Display Stream Compression，显示流压缩）**：VESA 定义的**视觉无损**压缩，把像素流压小后再传。带宽不够上 8K / 高刷时，靠 DSC 把数据塞进有限的 lane 里。**注意：一致性测试里，开不开 DSC、压缩比多少，是链路层要单独验证的一项。**

![图3：DP 的两大骨架。上半——时钟架构分水岭对比：HDMI TMDS = 3 数据 + 1 根独立时钟线转发（源同步，接收端不用 CDR）；DisplayPort = 只有数据 lane、时钟埋在数据里、接收端靠 CDR 捞（嵌入式，从 DP 1.1 起就是）。用一条竖虚线把两者分在"看时序 vs 看眼图/CDR"两岸，标注"DP 一直在右岸、HDMI 到 FRL 才过河"。下半——打包传输带来的能力：micro-packet 微包流 →①MST 菊花链（一根线 A→B→C 串多屏，各显各的）②DSC 视觉无损压缩（像素流压小塞进有限 lane，上 8K/高刷靠它）](/images/44/ill03_arch.png)

## 三、版本与速率：从 HBR 到 UHBR，编码换了、还加了 FEC

DP 的速率演进，可以按"编码方式"劈成两段：**8b/10b 的 HBR 时代**和 **128b/132b 的 UHBR 时代**。

| 版本 | 链路速率档 | 每 lane 速率 | 编码 | 4 lane 原始 / 有效带宽 |
|---|---|---|---|---|
| DP 1.1 | HBR | 2.7 Gbps | 8b/10b | 10.8 / ~8.64 Gbps |
| DP 1.2 | HBR2 | 5.4 Gbps | 8b/10b | 21.6 / ~17.28 Gbps |
| DP 1.3 / 1.4 | HBR3 | 8.1 Gbps | 8b/10b | 32.4 / ~25.92 Gbps（+DSC 上 8K） |
| DP 2.0 / 2.1 | UHBR 10 | 10 Gbps | 128b/132b | 40 / ~38.7 Gbps |
| DP 2.0 / 2.1 | UHBR 13.5 | 13.5 Gbps | 128b/132b | 54 / ~52.2 Gbps |
| DP 2.0 / 2.1 | UHBR 20 | 20 Gbps | 128b/132b | 80 / ~77.4 Gbps |

几个要点：

- **HBR / HBR2 / HBR3 用 8b/10b**，每 10 bit 里 8 bit 是数据，20% 开销——和 HDMI TMDS、USB 3.0 Gen1 一个套路。HBR3（8.1G/lane）是 8b/10b 时代的顶，配 DSC 才够上 8K。
- **UHBR（DP 2.0 起）换成 128b/132b**，开销从 20% 骤降到约 3%（效率 ~97%），和 USB4 / PCIe 6.0 之前那类思路一致。**UHBR20 四条 lane = 80 Gbps 原始、约 77.4 Gbps 有效**，是目前 DP 的天花板，够 **8K120 / 双 4K** 这种夸张规格。
- **UHBR 强制上 FEC（前向纠错）**：速率高到这个份上，误码难免，FEC 让接收端能**当场纠错、不用重传**。测试时 FEC 的行为也要单独验。

> ⚠️ 老规矩，**带宽别套错公式**（↔速查 02 / 第 29 篇）：UHBR20 的 20 Gbps/lane 是**串行波特率**，测眼图的示波器带宽按"约 0.7～1 倍波特率"选（十几到几十 GHz 档），**不是**拿"上升时间 5 倍谐波"去套出个吓人的两三百 GHz。串行高速看眼图，不是看方波谐波。

![图4：DP 版本与速率阶梯。横向时间轴从左到右：DP1.1 HBR 2.7G/lane → DP1.2 HBR2 5.4G → DP1.3/1.4 HBR3 8.1G（这三档统一 8b/10b、20% 开销，虚线框「8b/10b 时代」；HBR3 配 DSC 才够 8K）→ 一道竖分界线「换编码 + 加 FEC」→ DP2.0/2.1 UHBR10 10G / UHBR13.5 13.5G / UHBR20 20G（实线框「UHBR 时代：128b/132b、~3% 开销、强制 FEC」）。每档旁标 4 lane 总带宽：10.8/21.6/32.4G → 40/54/80G。右端大字标 UHBR20×4=80G 原始、约 77.4G 有效（8K120/双 4K）。底部一句：编码从 20% 开销降到 3%，是 DP 冲上 80G 的关键](/images/44/ill04_versions.png)

## 四、番外主角：eDP——你天天在用、却从没见过的那条 DP

用户最关心的"内屏"就在这里。**eDP（Embedded DisplayPort，嵌入式 DisplayPort）** 是 VESA 把 DP 裁剪、优化后专门用于**设备内部屏**的版本：笔记本、平板、一体机、很多手机的屏，主板到面板之间那条排线，跑的就是 eDP。

它和外接 DP 同源，但为"内屏、省电、省成本"做了几件专属的事：

- **PSR / PSR2 / Panel Replay（面板自刷新）**：屏里带一块帧缓存，**画面没变时，面板自己刷新自己**，GPU 和主链路就能休眠——静态桌面、看文档时**特别省电**。你笔记本"合盖不断电、屏一直亮却不怎么耗电"，就是它。
- **背光控制 / CABC / 局部调光**：eDP 能直接对面板背光下细粒度指令（外接 DP 显示器一般做不到这么细），按内容自适应调背光省电。
- **MSO（Multi-SST Operation，多单流切分）**：把一条链路切成多路，喂给高像素密度面板。
- **更省的电气档**：可以用更低摆幅、更少 lane 跑，一切为功耗让路。

**测试上的差别**：eDP 的一致性测试同样看眼图、看抖动，但**多了一堆"面板级"验证项**——PSR 进出退时序对不对、背光调光有没有闪、Panel Replay 局部刷新会不会撕裂。这些是外接 DP 没有、eDP 特有的坑。

![图5：eDP——设备内部那条看不见的 DP。左：主板 SoC/GPU ──eDP 排线──> 笔记本内屏面板（画出焊在里面、无插头，对比外接 DP 有插头）。右：eDP 专属四件事，各配一句测试关注点：①PSR/Panel Replay 面板自刷新（画面不变→面板自己刷、GPU 休眠省电｜测：进出退时序、局部刷新会不会撕裂）②背光控制/CABC 局部调光（按内容调背光｜测：调光闪烁 flicker）③MSO 多单流切分（喂高密度面板）④更低摆幅/少 lane（为功耗让路｜测：低摆幅下眼图还够不够）。底部一句：eDP=DP 内核 + 一堆面板级省电特性，测法在 DP 基础上多了"面板项"](/images/44/ill05_edp.png)

## 五、测什么·三类：Source 眼图、Sink 压力测试、线材通道

和第 38 篇的方法论、和 HDMI 一样，DP 一致性测试也落到**电气三阵营**，只不过因为 DP 一律嵌入式时钟，三类全程都在"CDR + 眼图"的世界里：

- **① Source / Tx（发送端）测试**：在 **TP2**（源输出经参考线缆 / 夹具后的点）抓**数据 lane 眼图**，扣 VESA 眼图模板，测抖动（RJ/DJ/TJ 分离↔32）、上升沿、电平摆幅、**pre-emphasis（预加重）各档**。HBR 一套模板、UHBR 又一套（更严、带参考均衡）。
- **② Sink / Rx（接收端）测试**：用 **BERT / 码型发生器**在 **TP3**（sink 输入点）造一个**校准过的"压力眼"**（loss + jitter 都加到规范上限），看 sink 还能不能正确接收——即**抖动容限（jitter tolerance）**、接收灵敏度、CDR/PLL 跟不跟得住（↔第 41 篇 Rx jitter tolerance，同一路数）。
- **③ Cable / 通道测试**：线材 / 连接器的插入损耗、回损、lane 间 skew、串扰——尤其 UHBR 线和 USB-C 线，通道裕量是过不过的关键。

![图6：DP 一致性测试三阵营（全程 CDR+眼图世界）。左 Tx/Source：示波器在 TP2 抓 4 条 data lane 眼图→扣 VESA 眼图模板+抖动分离(RJ/DJ/TJ)+预加重各档；标 HBR 模板 vs UHBR 模板(更严+参考均衡)。中 Rx/Sink：BERT/码型发生器在 TP3 造校准压力眼(loss+jitter 加满)→测抖动容限/接收灵敏度/CDR 跟踪(↔41)。右 Cable：VNA/示波器测插损/回损/skew/串扰，UHBR 线与 USB-C 线是重灾区。底部标 TP2=源端(过参考线缆)、TP3=宿端输入，两个测试点别搞混](/images/44/ill06_test3.png)

## 六、怎么测：进测试模式 + link training 到目标档 + 参考均衡

具体到台面上，DP 的流程和 HDMI FRL 高度相似（因为本来就是同一路）：

1. **搭夹具、接测试点**：Source 测试用带 TP2 探测点的夹具 / 转接；Sink 测试把码型灌到 TP3。
2. **通过 AUX / DPCD 让被测件进"测试模式"**：写 DPCD 寄存器，让 Source 固定输出规定的**测试码型（PRBS / 特定 pattern）**、锁定在目标速率档（比如 HBR3 或 UHBR20），关掉自适应，方便重复测量。
3. **link training 到目标档**：DP 传数据前，Source↔Sink 要先做 **link training**——从低速档往高试、调 lane 数 / 电压摆幅 / 预加重，训到一个稳定档位，结果写进 DPCD。训不上就回落低档（4K144 变 4K60 常死在这）。
4. **示波器抓眼图 / BERT 打压力**：Tx 侧抓眼图套模板；Rx 侧灌压力码型测容限。
5. **UHBR 必须上参考均衡**：UHBR 速率下裸眼图基本是"闭"的，**必须在示波器里挂上规范规定的参考均衡（CTLE / DFE↔34）**把眼睁开再判——这一步和 USB 3.x、PCIe 完全一个思路。

![图7：DP 测试台面搭建。左区块"进测试模式"：上位机经 AUX 写 DPCD 寄存器→被测 Source 固定输出 PRBS 测试码型、锁定目标速率档(HBR3/UHBR20)。中区块"link training"：Source↔Sink 从低速往高试→调 lane 数/摆幅/预加重→训到稳定档写回 DPCD→训不上则回落(4K144→4K60)。右区块"测量"：TP2 接示波器抓眼图套 VESA 模板；TP3 接 BERT 灌压力眼测容限。底部强调框：UHBR 必挂参考均衡 CTLE/DFE(↔34) 才能把闭着的眼睁开判读——和 USB3/PCIe 同路](/images/44/ill07_bench.png)

## 七、⚙️ 关键仪器设置：把模块三的 SI 概念落到旋钮上

DP 的仪器设置几乎照搬 HDMI FRL / USB 3.x 那套（这也是"学会一个通一片"的好处）：

- **去嵌（de-embed）↔第 33 篇**：把夹具 / 探测通道的损耗从测量结果里数学扣掉，还原芯片管脚处的真实信号。
- **嵌入参考线缆 / 参考通道**：Source 测试要在"最坏情况参考线缆"后测（TP2 的定义），模拟真实线材损耗。
- **参考均衡 CTLE / DFE↔第 34 篇**：UHBR 强制用，把高频损耗补平、把闭着的眼睁开——**判据是"参考均衡之后"的眼图**，不是裸眼图。
- **时钟恢复 golden PLL↔第 32 篇**：DP 是嵌入式时钟，示波器要用规范规定的黄金 PLL 环路参数恢复时钟，才能正确测抖动。
- **带宽按规范档设↔第 29 / 速查 02**：按波特率约 0.7～1 倍选示波器带宽，别过冲。
- **抖动分离 + eye mask↔第 31 / 32 篇**：把 TJ 拆成 RJ / DJ，再套眼图模板判裕量。

> 一句话：**DP（尤其 UHBR）的仪器设置 = USB 3.x / PCIe 那套嵌入式时钟测法**。你在第 41 篇学的东西，这里几乎能直接复用；HDMI 的 TMDS 那半套（转发时钟）在 DP 上完全用不到。

## 八、判据从哪来：眼图模板 / 抖动预算不是拍脑袋

和第 38 篇讲的一样，DP 的判据也来自 **VESA 的 CTS（Compliance Test Specification，一致性测试规范）**，不是拍脑袋：

- **眼图模板、抖动限值**：从信道损耗预算 + 抖动预算反推而来（↔第 38 篇）。HBR 系列一套、UHBR 系列一套（后者含"参考均衡之后"的判定）。
- **预加重 / 电压摆幅各档**：每个组合都要落在规范窗口内。
- **link training 结果 + FEC 行为**：链路层 CTS 要验训练能不能到目标档、FEC 纠错行为对不对。
- **DSC / MST 单独验**：开压缩、开多流时的正确性，是链路层的独立测试项。

DP 的判据体系比 HDMI 更"分层清楚"：**PHY CTS（电气：眼图 / 抖动 / TP2 TP3）+ Link Layer CTS（协议：训练 / DPCD / EDID / MST / FEC / DSC）** 两大本，分别由不同测试系统跑。

## 九、常见翻车与调试：黑屏 / 掉刷新率 / 菊花链带不动 / eDP 闪，怎么分层揪

DP 翻车，**先分层、别一上来就赖信号**（顺序和 HDMI 一样，只是多了 MST / DSC / eDP 几层）：

1. **先看协商层（AUX / DPCD / EDID）**：屏认不认得出？EDID 读到没？DPCD 里 link training 训到第几档？——**"高刷选项灰掉""插上没反应"多半死在这**，跟信号无关。
2. **再看电气层（眼图 / 均衡 / 线材）**：训到高档但花屏、闪断、雪花——量数据 lane 眼图、skew、看线材够不够 UHBR 级、参考均衡后眼还睁不睁。
3. **压缩 / 多流层（DSC / MST）**：8K 上不去 / 开 DSC 才行；菊花链第二台屏黑——查 MST 的流分配、Hub 带宽是否够。
4. **内容保护层（HDCP）**：黑屏但灯亮、放版权内容才黑——HDCP 握手问题，和信号质量无关。
5. **eDP 专属层**：内屏花屏 / 闪烁 / 撕裂——查 PSR 进出退时序、背光调光、Panel Replay 局部刷新（外接 DP 没有这层）。

**现象速查**：高刷选项灰掉=EDID/DPCD 没协商上；训到高档仍花屏雪花=眼图/线损/skew；菊花链第二屏黑=MST 流分配/Hub 带宽；8K 只能开 DSC 才行=带宽到顶；黑屏灯亮放版权内容才黑=HDCP；笔记本内屏闪烁/撕裂=eDP 的 PSR/背光/Panel Replay。

## 一句话总结

**HDMI 是在分水岭上走了个来回，DisplayPort 从出生就站在"嵌入式时钟 + CDR + 打包传输"这一岸——所以它的一致性测试一上来就是 USB3 / PCIe 那套：看眼图、做抖动分离、用参考均衡、TP2/TP3 分点，到 UHBR（128b/132b + FEC）把裕量压到极限；再加上笔记本里那条 eDP 特有的面板级省电项，构成了 DP 测试和 HDMI 既重叠又不同的那部分。学会了 HDMI FRL，DP 你已经会了一大半。**
