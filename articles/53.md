先说个接串口模块、调试打印、连 GPS / 蓝牙 / 4G 模组的人几乎都撞过的南墙。

你把一个模块（GPS、蓝牙、Wi-Fi、4G 模组，或者就是板子的调试串口 debug uart）接到 MCU / 电脑的串口上，打开串口助手一看：**收回来全是乱码**；又或者，**9600 波特率好好的，一提到 115200 就开始乱**；再或者，**接上去什么都收不到**，查了半天发现 **TX 和 RX 没交叉**（两个都是 TX 对 TX）；还有更惨的——**把一个 RS-232 的设备（±12V）直接怼到 MCU 的 TTL 脚（3.3V）上，脚直接烧了**。

做硬件、写嵌入式的人慢慢会明白：**UART 这类「乱码、提速就乱、接上没反应、烧脚」，绝大多数根本不是模块坏、也不是芯片坏，而是这几件约定 / 物理层的事没对上——两端波特率对没对准、帧格式（数据位 / 校验 / 停止位）一不一样、TX/RX 有没有交叉、有没有共地、电平（TTL / RS-232 / RS-485）兼不兼容。** 波特率差得多，接收端的采样点越数越偏，一帧还没数完就采到隔壁位上，读出来自然乱码；帧格式设错（比如一端 8N1、一端 8E1），校验位当数据位、停止位对不上，整片错；TX 对 TX、没共地，信号根本回不来或没有参考。

UART 到底怎么工作、为什么"没有时钟线还能通"、波特率和采样点是怎么回事、它的信号和时序该怎么测、为什么判据在"两端约定"而不在一张规范表——这，就是这一篇要讲的，也是我们**低速板级总线这条线的第三站、也是收官对比的一站**。

## 一、先认清主角：UART 是什么、它和 I2C / SPI 到底不一样在哪

**UART**（Universal Asynchronous Receiver / Transmitter，通用异步收发器）严格说是芯片里那个**收发外设**，我们平时说的"串口 / COM 口 / 串行通信"，指的就是它跑的这套**异步串行**协议。满世界都是它：**调试打印（printf 到串口）、GPS / 蓝牙 / Wi-Fi / 4G 模组、各种传感器和仪器的串口、单片机之间的通信、工业现场的 RS-485 总线**——凡是"两根线、不想布时钟、点对点或一主多从慢速传数据"的场合，很多都走 UART。

它最少只用**两根信号线**（外加地）：

- **TX（Transmit，发送）**：本机往外发数据的线。
- **RX（Receive，接收）**：本机接收数据的线。
- **接线要交叉**：A 的 **TX 接 B 的 RX**，B 的 **TX 接 A 的 RX**——这样各自发、各自收，是**全双工**。**别忘了还要共地（GND）**，否则两端没有共同的电压参考，电平无从判断。

它和前两篇 I2C、SPI，有几个根本区别——**每一个都直接决定了它怎么测**：

1. **异步、没有时钟线（最根本的一条）**：I2C 有 SCL、SPI 有 SCLK，接收方跟着这根**共享时钟**的节拍采样，主机说了算。**UART 一根时钟线都没有**——收发两端各用各的本地时钟，**事先约定好同一个波特率**（比如 115200 bit/s，每位约 8.68µs），发送方按这个节奏一位一位吐，接收方**先靠一个"起始位"对齐，然后按约定的位宽，用自己的时钟去数、在每一位的正中间采样**。没有时钟线对齐，就意味着**两端的波特率必须足够接近**，差多了采样点会一路漂偏——这是 UART 测试第一个、也是最独特的主角。

2. **靠"起始位"帧同步，不是靠时钟沿**：线路空闲时是**高电平**。发送方要发一个字节，先拉低一位（**起始位 start bit**），这个**下降沿**就是给接收方的"发令枪"——接收方一看到线从高变低，就知道"一帧来了"，从这个边沿开始计时、按波特率去采后面每一位。**帧的对齐点是起始位，不是某根时钟线的沿**。

3. **点对点、无主从、无地址、无 ACK**：I2C 靠地址点名、SPI 靠片选选设备；**UART 最基本的形态就是两个设备一对一直连**，没有主机 / 从机之分（谁都能主动发）、没有地址、也没有 ACK 应答。要一主多从？那是**物理层升级到 RS-485** 之后、再在**上层协议（如 Modbus）里加地址**才有的事，UART 本身不管。

4. **全双工，但和 SPI 的全双工不是一回事**：SPI 的全双工是**主机出一个时钟沿，两根数据线同时收发一位**，收发是**同步咬合**的。UART 的 TX、RX 两根线**各自独立、异步**，发和收**互不相干、想发就发**——没有"每个时钟沿交换一位"那种咬合关系。

5. **判据不在一张规范表，也不在一本时序手册，而在"两端约定 + 物理层电平标准"**：I2C 有 NXP UM10204 统一表，SPI 是一颗芯片一本 datasheet 的时序。**UART 的核心判据是"约定"**——波特率、数据位、校验、停止位，**两端设成一样、且波特率累计误差在预算内（通常 ±2%~±3%）**，就能通；**物理层的电平判据**则看你走哪种：**TTL** 看两端芯片手册的 V_IH / V_IL，**RS-232 / RS-485** 各有 **TIA/EIA 标准**（电压 / 差分门槛 / 端接）可查。

一句话：**UART = 异步、无时钟线、TX/RX 两线交叉、靠约定波特率各自采样、点对点、全双工的串行口**；它测的是**波特率误差 + 帧格式 + 采样点 + 协议解码**（外加物理层的电平 / 差分），把 I2C / SPI「时钟线 + 采样沿」那套彻底扔了。

## 二、原理：异步无时钟怎么通——起始/数据/校验/停止一帧、波特率与采样点

UART 的原理，核心就一件事：**没有时钟线，怎么让接收方知道"每一位在哪、什么时候采"？** 答案拆成三块，每一块都直接对应一类要测的东西。

### 2.1 异步、无时钟：靠"约定波特率 + 起始位"各自采样

同步总线（I2C/SPI）是"我给你时钟，你跟着我的沿采"；**异步的 UART 是"我们事先约好每位多宽，然后你自己数着采"**。

- **波特率（baud rate）**：约定的**每秒传多少位**，比如 **115200 baud = 每位约 8.68µs**（1/115200 秒）。常见还有 9600、19200、38400、57600 等。**两端必须约成同一个数**。
- **线路空闲 = 高电平**。要发一个字节，发送方先把线**拉低一位的时间**——这就是**起始位（start bit）**。这个**从高到低的下降沿**，是给接收方的**发令枪**：接收方一直盯着 RX，一看到下降沿，就知道"一帧开始了"，**从这一刻起，按约定的波特率（每位多宽）去计时，在每一位的正中间采样**。
- 数据发完，发送方把线**拉回高电平至少一位**——这就是**停止位（stop bit）**，让线回到空闲、也给下一帧留出可辨认的起始沿。

**关键点来了**：接收方是用**自己的本地时钟**去数"每位 8.68µs"的。如果两端波特率有偏差（比如一端实际是 115200、另一端因为分频凑不出、实际是 118000），那么**接收方每采一位就偏一点，误差逐位累积**。等数到一帧的最后一位，采样点已经偏出这一位的范围、跑到隔壁位上了——**这一位、乃至整帧就采错，表现为乱码**。所以：**没有时钟线的代价，就是"波特率必须足够准"，这也是 UART 最独特的测点。**

### 2.2 帧格式：起始 + 数据 + 校验 + 停止，8N1 是什么意思

UART 一帧（发一个字符）的结构固定是这几段，**两端必须约定成完全一样**：

| 段 | 长度 | 说明 |
|---|---|---|
| **起始位 start** | 1 位（低） | 发令枪，线从空闲高拉低，标志一帧开始 |
| **数据位 data** | **5~9 位，通常 8 位** | 真正的数据，**LSB 先发**（低位在前） |
| **校验位 parity** | 0 或 1 位 | 可选：无（N）/ 偶校验（E）/ 奇校验（O），做简单查错 |
| **停止位 stop** | **1 / 1.5 / 2 位（高）** | 帧结束，线回到空闲高，给下一帧留起始沿 |

- **常写成 "8N1"**：**8** 个数据位、**N**o parity（无校验）、**1** 个停止位——这是最常见的格式。**8E1** 就是 8 数据 + 偶校验 + 1 停止；**7E1**、**8N2** 等以此类推。
- **数据位 LSB first**：UART 先发最低位。解码 / 逻辑分析仪里 bit 顺序设反了，也会读成"乱码"。
- **校验位**：偶校验让"数据位 + 校验位里 1 的个数"为偶数（奇校验则为奇数），接收方一算不对就报**校验错（parity error）**。它只能查出奇数个位翻转，能力有限，但能挡掉一部分噪声误码。
- **停止位对不上**：接收方在该看到"高停止位"的地方看到的是低，就报**帧错（framing error）**——这常常正是**波特率偏太多**或**帧格式设错**的现场信号。

所以：**一帧 8N1 = 1 起始 + 8 数据 + 1 停止 = 10 位。** 传一个字节，线上实际要跑 10 位——这也是为什么"115200 波特率"折算成字节率大约是 11520 字节/秒，而不是 14400。

### 2.3 采样点与过采样：接收端怎么"采在每一位正中间"

接收方怎么保证采样点落在每一位的**正中间**（离两边的跳变最远、最稳）？靠**过采样（oversampling）**：

- 接收方内部的时钟通常是波特率的 **16 倍（16× 过采样）**（有的用 8×）。也就是说，**每一位的时间里，它偷偷采 16 个小样**。
- 检测到起始位下降沿后，它**先数到第 8 个小样（半位处）确认起始位仍是低**（滤掉毛刺），然后**每隔 16 个小样、在每位的中点（第 8 个附近）采一次**；讲究点的会取**中间第 7 / 8 / 9 三个样做多数表决**，抗噪。
- **为什么这样能容忍一点波特率误差**：因为采样点定在**每位正中间**，离左右两条位边界各有半位的余量。只要两端波特率误差累积到**最后一位时，采样点还没漂出那半位的余量**，就还采得对。**一旦漂出去，最后几位先错——这就是"帧尾比帧头更容易错"的原因**。

这也解释了 **UART 的波特率误差预算**：一帧 10 位、采样在中点，理论上最后一位允许的累计偏差约半位（对 10 位帧约 ±5% 的极限），但要扣掉**接收方自己 16× 量化的粒度（约 ±1/16 ≈ 6.25%/位）** 和裕量，**工程上通常要求两端波特率累计误差 ≤ ±2%~±3%**。这就是为什么"9600 能通、115200 就乱"——**高波特率下，分频器凑不出精确的数、误差变大，就顶穿了预算**。

![图4：UART 帧格式与波特率原理(对应 §2)。上：一帧波形——线空闲高→起始位拉低 1 位(下降沿=发令枪)→8 个数据位 LSB 先发→可选校验位→停止位拉回高≥1 位→回空闲。标 8N1=1 起始+8 数据+1 停止=10 位。中：波特率=每秒多少位，115200=每位约 8.68µs，两端必须约成同一个数；线上跑 10 位才传 1 字节。下：帧格式表 起始 1 低/数据 5~9 通常 8 位 LSB 先/校验 0-1 位 N-E-O/停止 1-1.5-2 位高；停止位对不上=帧错 framing error、校验不符=parity error。一句话：没有时钟线，靠约定波特率+起始位下降沿对齐，两端波特率必须够准](/images/53/ill03_frame.png)

### 2.4 两线 / 四线 / 六线：硬件流控 RTS/CTS 是怎么回事（一个高频真坑）

前面说 UART 最少两根信号线（TX/RX + 地）。但你在模组数据手册、串口助手的设置里，一定见过 **"流控 Flow Control：None / Hardware(RTS/CTS) / Software(XON/XOFF)"**，也常听到 **"四线 UART"** 这个说法——这里一次说清：

- **两线 UART**：TX / RX（+ GND）。最常见，没有流控，发送方只管往外吐。
- **四线 UART = TX / RX + RTS / CTS 两根硬件流控握手线**。**RTS**（Request To Send，请求发送）、**CTS**（Clear To Send，允许发送）：现代嵌入式里，**本机的 RTS 输出交叉接到对端的 CTS 输入**（和 TX/RX 一样交叉）。机制是——**接收方 FIFO 快满了，就把 RTS 置为"别发了"，对方看到自己的 CTS 无效就暂停发送；接收方处理完、腾出空间，再把 RTS 置回"可以发"，对方继续**。全程硬件自动完成、不占数据带宽。它的作用只有一个：**防止高波特率 / 突发大数据时，接收方来不及处理、FIFO 溢出丢字节**。
- **六线（及以上）**：再加 **DTR/DSR**（数据终端 / 数据设备就绪）等 modem 控制线，是老 RS-232 那套完整握手（DTE/DCE），现在嵌入式里基本用不到。
- **软件流控 XON/XOFF**：不加线，靠在数据流里插两个特殊字节（XOFF=0x13 喊停、XON=0x11 继续）。省线，但会占用这两个字节、对**二进制数据不友好**（数据里正好出现 0x11/0x13 就误触发），所以传纯文本才用。

> ⚠️ **一个和上面"乱码"完全不同的病**：如果现象是 **"平时好好的，一到大数据量 / 高波特率就偶尔丢几个字节、或整段丢"**，而**不是整片乱码**——十有八九是**没开硬件流控**（或一端开了一端没开、或 RTS/CTS 没接 / 接反），接收方 FIFO 溢出把数据顶掉了。这和"波特率不匹配导致的整片乱码"是**两个不同的故障**，别混。有些模组（如**蓝牙 HCI 接口、部分 4G 模组**）在高波特率下**强制要求硬件流控**，不接不开轻则丢数据、重则初始化都过不去。**测法**：逻辑分析仪 / 示波器**多抓 RTS、CTS 两根线**，看丢数据的瞬间接收方 RTS 有没有及时拉去喊停、发送方 CTS 被拉停后有没有真的停下；同时确认 **RTS↔CTS 交叉接对**、且**两端串口配置里都开了 Hardware 流控**。

## 三、物理层电平变体：TTL UART / RS-232 / RS-485 / RS-422（UART 的"家族"）

**这里要划一条重要的界线：UART 是"协议 / 逻辑层"（帧格式、波特率那套），而线上到底用多高的电压、单端还是差分、能拉多远，是"物理层"的事。** 同一套 UART 帧，可以套在不同的物理层上跑——这就是 UART 的"家族"，也是最容易接错、烧脚的地方：

| 变体 | 电平 / 方式 | 逻辑 | 拓扑 | 距离 | 典型用途 |
|---|---|---|---|---|---|
| **TTL UART** | 单端，0 / Vdd（3.3V 或 5V） | 空闲高、起始低（正逻辑） | 点对点 | 板级 / 短线（几十 cm） | MCU 引脚直连、板内模块、调试串口 |
| **RS-232** | 单端，**双极性 ±3~±15V** | **反相**：mark(1)=负、space(0)=正 | 点对点 | 十几米 | 老 PC 串口、仪器、工控 |
| **RS-485** | **差分 A/B** | 差分门槛（\|A-B\|>±200mV） | **多点（一主多从，32+ 节点）** | **上千米** | 工业总线、Modbus、楼宇 |
| **RS-422** | **差分** | 差分门槛 | 一发多收（点对多点） | 上千米 | 长距离点对点 / 一对多 |

几个要命的关键点：

- **TTL 和 RS-232 电平完全不同、逻辑还是反的**：TTL 是 0/3.3V、空闲高；**RS-232 是 ±12V 上下摆、而且逻辑反相**（空闲是负电平）。**两者绝对不能直连**——把 RS-232 的 ±12V 直接怼进 3.3V 的 MCU 脚，**分分钟烧脚**。中间必须加**电平转换芯片（如 MAX232）**做电平 + 反相转换。这就是开头"烧脚"那一幕的真相。
- **RS-485 / RS-422 是差分、能拉很远、抗干扰强**：用 **A、B 两根线的电压差**表示 0/1，共模噪声两根线一起抬、差分不受影响，所以能在**上千米、强干扰的工业现场**跑。代价是要加**收发器芯片**（如 MAX485）、**端接电阻（两端各 120Ω，匹配线缆特性阻抗）**、**偏置电阻**，长线高波特率还要管**反射**——这时候，模块一 SI 那套（特性阻抗、端接、反射）**又回来了**，只是发生在差分线缆上。
- **RS-485 是半双工的多点总线**：最常见的 **2 线 RS-485** 是**半双工**——同一对 A/B，收发分时，靠一个**方向控制脚（DE/RE）** 切换谁在发。**方向切换的时机（什么时候使能发送、发完多久才松开）是 RS-485 特有、极爱翻车的时序测点**：松早了把自己最后一位截断，松晚了挡住对方的回复。4 线 RS-485 才是全双工。
- **UART 帧不变，变的只是物理层**：不管 TTL、RS-232 还是 RS-485，**上面跑的都是同一套"起始 / 数据 / 校验 / 停止 + 约定波特率"**。所以协议解码时，逻辑是一样的；**只有电平判据、要不要反相、要不要差分探头 / 端接，随物理层变**。

![图6：UART 物理层电平变体(TTL/RS-232/RS-485/RS-422 家族，对应 §三)。四行：①TTL UART=单端 0/Vdd(3.3 或 5V)·正逻辑空闲高起始低·点对点·板级短线·MCU 直连/板内模块/调试串口②RS-232=单端双极性±3~±15V·反相 mark1=负 space0=正·点对点·十几米·老 PC 串口/仪器/工控③RS-485=差分 A/B(|A-B|>±200mV)·多点一主多从 32+节点·上千米·工业 Modbus④RS-422=差分·一发多收·上千米·长距离。底部三个要命点：TTL 与 RS-232 电平完全不同且逻辑反相绝不能直连(±12V 怼 3.3V 脚烧脚)，中间加 MAX232 转换；RS-485/422 差分能拉远抗干扰，要收发器 MAX485+两端 120Ω 端接+偏置，长线高波特反射回来(模块一 SI)；RS-485 2 线半双工靠 DE/RE 方向脚切换，切换时机极爱翻车。UART 帧不变，变的只是物理层——协议解码逻辑一样，电平/反相/差分/端接随物理层变](/images/53/ill05_levels.png)

## 四、独特之处：无时钟靠波特率对齐 + 起始位帧同步 + 点对点无寻址 + 物理层电平多变

UART 有四处和 I2C / SPI 很不一样、直接影响测试的独特设计：

- **① 无时钟线，波特率就是命门**：I2C / SPI 有共享时钟兜底，采样点永远对得上；**UART 没有时钟线，采样点全靠"两端波特率约得准 + 各自本地时钟数得准"**。所以 **UART 测试的第一主角，是"量出线上实际波特率、算两端误差有没有超预算"**——这是 I2C / SPI 完全没有的测点。误差从哪来？多半是**波特率分频器凑不出精确值（时钟源除不尽）**，高波特率下尤其明显。

- **② 靠起始位帧同步、无时钟沿**：帧的对齐点是**起始位那个下降沿**，不是某根时钟线。所以**线上的毛刺 / 噪声若被误当成起始沿**，接收方就会在错误的时刻"开数"，整帧全错——这也是为什么**空闲电平的干净、起始沿的陡直**对 UART 有意义（尤其长线 RS-485）。

- **③ 点对点、无主从、无地址、无 ACK**：最基本的 UART 就是两个设备对发，**没有片选、没有地址、没有应答**。所以协议层极简：**看的就是"帧本身对不对"**，而不像 I2C 有"地址 / ACK"、SPI 有"CS 帧"。要多点，得靠 **RS-485 物理层 + 上层协议（Modbus 等）自己加地址和收发方向控制**。

- **④ 物理层电平多变，接错就出大事**：I2C / SPI 基本都是板级同电平（顶多加个电平转换）；**UART 却可能是 TTL、RS-232（±12V 反相）、RS-485（差分）中的任意一种**。**电平不兼容轻则收不到、重则烧脚**；差分的 RS-485 还要用**差分探头**、管**端接和方向时序**。测 UART 前，**先搞清楚这条线是什么物理层**，比什么都重要。

![图7：UART 四大独特(对照 I2C/SPI)。①无时钟线波特率是命门：I2C/SPI 有共享时钟兜底采样点永远对得上，UART 全靠两端波特率约得准+本地时钟数得准，第一主角=量线上实际波特率算两端误差超没超预算(I2C/SPI 没有的测点)，误差多来自分频器凑不出精确值高波特率尤甚。②起始位帧同步无时钟沿：对齐点是起始位下降沿，线上毛刺被误当起始沿→整帧全错，所以空闲干净/起始沿陡直有意义(长线 RS-485 尤其)。③点对点无主从无地址无 ACK：两设备对发，协议极简只看帧本身对不对，多点要 RS-485+上层协议 Modbus 加地址和方向。④物理层电平多变接错出大事：可能 TTL/RS-232 ±12V 反相/RS-485 差分任意一种，不兼容轻则收不到重则烧脚，差分要差分探头+端接+方向时序，测前先搞清物理层](/images/53/ill06_unique.png)

## 五、测什么 · 三类：信号电气、时序（波特率 / 帧）、协议解码

UART 的测试，也归成三大类，**但主角从 I2C 的"上升时间"、SPI 的"采样沿模式"，换成了"波特率误差 + 帧格式 + 采样点"**：

1. **信号电气（Signal / Electrical）**：用示波器看 TX / RX 波形，量：**高低电平与逻辑门槛裕量（够不够被识别）、边沿、空闲电平干不干净、有没有毛刺**。物理层不同重点不同——**TTL** 量 0/Vdd 够不够门槛；**RS-232** 量**双极性 ±电平**是否达标、逻辑是否反相；**RS-485** 要用**差分探头**量 **\|A-B\| 差分幅度、共模电平**，长线高波特还要看**反射 / 振铃、端接对不对**。UART 本身慢，SI 通常不像 SPI 那么吃紧，**但 RS-485 长线高波特是例外**。

2. **时序（Timing）——UART 的重头戏**：这是 UART 最独特的一类。核心是**量线上实际的"位宽"，反推实际波特率，再算两端误差**：
   - **量一位的宽度**：抓一段波形，量**最窄的那个电平段（就是一位的时间）**，`实际波特率 = 1 / 位宽`。最方便的办法是**发 0x55（二进制 01010101）**——它在线上是**完美的方波，每半周期正好一位**，一量周期就出波特率（**0x55 是业界校波特率的经典字节**）。
   - **算两端误差**：把量到的实际波特率和约定值比，`误差% =（实际−约定）/ 约定 × 100%`，**两端各自的误差 + 累计，要在 ±2%~±3% 预算内**。
   - **量帧结构**：起始位宽、每个数据位宽、停止位在不在、够不够宽；**看采样点（每位中点）会不会因误差漂出位边界**。
   - **RS-485 还要量方向切换时序**：DE/RE 使能相对第一位 / 最后一位的时机。

3. **协议解码（Protocol）**：用示波器的 **UART 触发 / 解码**或**逻辑分析仪**，把波形翻成**字节 / ASCII**：**设对波特率、数据位、校验、停止位、bit 顺序（LSB first）、是否反相（RS-232）**，然后看解出来的字节对不对、有没有 **framing error / parity error**。这一类回答"收到乱码（波特率？格式？反相？）、偶尔错几个字节（波特率临界 / 噪声）、完全收不到（TX-RX 没交叉 / 没共地 / 电平不对）"。

贯穿三类的度量，和前两篇差别很大：**没有时钟线可参考、没有采样沿模式、没有片选 / 地址**；换成了**线上实际波特率与误差、帧格式匹配、采样点位置**。和高速串行比，它**依然没有眼图、没有抖动分离、不用去嵌、不用 BERT**——还是低速那一岸（RS-485 差分也只是慢速差分，不上眼图）。

![图8：UART 测什么三类(主角=波特率误差+帧格式+采样点，不是上升时间也不是采样沿模式)。①信号电气：示波器看 TX/RX 电平与门槛裕量、边沿、空闲干不干净有无毛刺；TTL 量 0/Vdd 够不够门槛、RS-232 量双极性±电平+是否反相、RS-485 用差分探头量|A-B|差分幅度+共模+长线高波特看反射振铃端接；UART 慢 SI 通常不吃紧但 RS-485 长线例外。②时序(重头戏)：量最窄电平段=一位宽→实际波特率=1/位宽(发 0x55 方波最方便，经典校波特率字节)→算两端误差(实际−约定)/约定 要在±2~3%内；量起始/数据/停止位宽、采样点会不会漂出位边界；RS-485 量 DE/RE 方向切换时序。③协议解码：UART 触发解码/逻辑分析仪设对波特率/数据位/校验/停止/bit 顺序 LSB/是否反相→翻字节 ASCII，看 framing/parity error，回答乱码/偶尔错/收不到。共同：无时钟无采样沿无片选地址；仍无眼图抖动去嵌 BERT](/images/53/ill07_test3.png)

## 六、怎么测 · 台面：示波器量位宽算波特率 + UART 解码 / 逻辑分析仪 + 查交叉共地电平

UART 的测试台面，比 SPI 还**朴素**（单端一根线就能看一个方向），但有它**独特的两步**（量波特率、认物理层）：

- **① 一两个通道就够，先认物理层**：UART 一个方向就一根线，看 TX（或 RX）一个通道即可，要同时看收发就两个通道。**动手前先确认这条线是 TTL / RS-232 / RS-485**——**RS-232 别拿单端探头当 TTL 量**（它是 ±12V），**RS-485 必须用差分探头**量 A-B。搞错物理层，量出来的电平全是错的、甚至可能碰坏探头 / 板子。

- **② 发 0x55 量波特率（最实用的一招）**：让待测端**连续发 0x55**（`0101 0101`），线上就是一串**规整方波**，示波器量**周期 / 半周期**——半周期 = 一位时间，`实际波特率 = 1 / 位宽`。把它和约定值比、算误差。**没有 0x55 就抓任意一帧量最窄的那个电平段**。这是判断"波特率对不对"的最快、最硬的证据。

- **③ 量帧、对格式**：抓一帧完整波形，数**起始 1 位 + 数据几位 + 有没有校验 + 停止几位**，和你设的格式（8N1？8E1？）对上；停止位该高的地方是不是高。**格式对不上，解码器解出来也是乱码。**

- **④ UART 解码 / 逻辑分析仪，设对全部参数**：打开示波器 **UART 解码**或接**逻辑分析仪**，**波特率、数据位、校验、停止位、bit 顺序（LSB first）、是否反相（RS-232 要勾 invert）** 一个都不能设错，然后看解出来的字节 / ASCII 对不对、有没有 framing / parity error。**乱码先怀疑波特率，其次帧格式 / 反相。**

- **⑤ 查物理连接（收不到时第一步）**：**TX 是不是接到对端 RX（交叉）、有没有共地 GND、电平兼不兼容**（TTL↔RS-232 之间有没有 MAX232 转换、电压域对不对）。**"完全收不到"十有八九是这一层**，而不是波特率。

- **⑥ RS-485 专项**：接**端接（两端各 120Ω）** 了吗、有没有**偏置电阻**、**方向控制脚 DE/RE 的切换时机**对不对（用示波器同时看 DE 和数据线，确认发完最后一位再松开、松开够快不挡对方回复）、长线高波特看**差分波形有没有反射 / 振铃**。

![图9：测 UART 的台面(比 SPI 更朴素单端一根线，但有量波特率+认物理层两步独特)。①一两通道够+先认物理层：一个方向一根线看 TX/RX，动手前确认 TTL/RS-232/RS-485——RS-232 别拿单端当 TTL 量(±12V)、RS-485 必须差分探头量 A-B，搞错电平全错甚至碰坏探头。②发 0x55 量波特率(最实用)：连续发 0x55=规整方波，量半周期=一位宽→实际波特率=1/位宽，和约定比算误差；没 0x55 抓任意帧量最窄电平段。③量帧对格式：数起始 1+数据几位+校验?+停止几位，和 8N1/8E1 对上，停止位该高是否高。④UART 解码/逻辑分析仪设对全参数：波特率/数据位/校验/停止/bit 顺序 LSB/是否反相(RS-232 勾 invert)，看字节 ASCII+framing/parity error，乱码先怀疑波特率再格式反相。⑤查物理连接(收不到第一步)：TX 接对端 RX 交叉/有没共地 GND/电平兼容(TTL↔RS-232 有没 MAX232)。⑥RS-485 专项：两端 120Ω 端接+偏置+DE/RE 方向切换时机+长线差分反射振铃](/images/53/ill08_bench.png)

## 七、判据：在"两端约定"和"物理层电平标准"里，不在统一规范表也不在眼图

UART 的判据，和 I2C、SPI、和高速串行都不一样，最需要转过弯来：

- **I2C**：查 **NXP UM10204** 一张**统一规范表**——各档速率、电平、时序印死。
- **SPI**：**一颗芯片一本 datasheet** 的时序（f_SCK、setup/hold、CS）。
- **高速串行（模块六）**：从信道损耗**反推眼图模板 / 抖动预算**。
- **UART**：判据**一半在"约定"、一半在"物理层标准"**：
  - **协议 / 时序判据 = 两端的"约定"必须一致，且波特率误差在预算内**：波特率、数据位、校验、停止位**两端设成完全一样**；**两端波特率累计误差 ≤ ±2%~±3%**（一帧 10 位、中点采样能容忍的极限，扣掉裕量后的工程值）。这不是查某张表，而是**你和对端约好、并实测验证**的。
  - **电平判据 = 看物理层各自的标准**：**TTL** 看**两端芯片手册的 V_IH / V_IL**（能不能被识别、电压域配不配）；**RS-232** 看 **TIA/EIA-232** 标准的**双极性电压范围（±3~±15V、门槛 ±3V）**；**RS-485 / RS-422** 看 **TIA/EIA-485/422** 的**差分门槛（±200mV）、共模范围、端接（120Ω）**。这些物理层是**有标准可查的**，和"波特率约定"那半正好互补。

所以测 UART 的判据动作就两句：**① 波特率 / 帧格式，两端约的必须一样，实测波特率误差压在 ±2~3% 内；② 电平按你走的物理层（TTL 看手册、RS-232/485 看 TIA 标准）逐项对。**

一句话：**UART 没有"一张表管所有"——判据一半是"你和对端约好的那组参数（波特率 / 帧格式）实测对不对得上"，一半是"物理层（TTL/RS-232/RS-485）各自的电平标准"。**

## 八、常见翻车 → 分层揪真凶（别一上来就怀疑模块坏）

UART 的"乱码、提速就乱、收不到、烧脚"，也最忌一上来换模块、换芯片。正确姿势还是**分层**：

- **① 物理连接层（"完全收不到"最高频）**：**TX/RX 没交叉**（两个都 TX 对 TX）、**没共地 GND**、**电平不兼容**（TTL 3.3V 直连 RS-232 ±12V——轻则收不到、重则**烧脚**）。**"一个字节都收不到"先查这一层**，而不是调波特率。动手前用万用表 / 示波器确认线接对、有共地、电平域一致。

- **② 波特率 / 采样层（"乱码、提速就乱"最高频）**：**两端波特率没对准**——发 0x55 量实际波特率，算误差超没超 ±2~3%；典型「9600 通、115200 乱」= 高波特率下**分频器凑不出精确值**，误差顶穿预算，采样点漂到隔壁位。对策：**换一个两端时钟都能精确分频的波特率**、**改晶振 / 调分频**、或**降波特率**。

- **③ 帧格式层（"整片乱码 / 偶发校验错"）**：**数据位 / 校验 / 停止位两端不一致**（一端 8N1、一端 8E1，或 bit 顺序、反相设反）——解码 framing error / parity error 频发。核对两端**数据位、校验、停止位、LSB first、RS-232 是否反相**是否完全一致。

- **④ 信号 / 物理质量层（长线 RS-485 才犯）**：**RS-485 长线高波特没端接 / 端接不对 → 差分反射振铃 → 误码**；**方向控制 DE/RE 时机不对 → 截断自己最后一位或挡住对方**；**共模干扰超范围**。量差分波形、查 120Ω 端接和偏置、看 DE 切换时机（呼应模块一 SI + RS-485 专项）。

**口诀**：收不到先查交叉 / 共地 / 电平（物理连接）→ 乱码先发 0x55 量波特率算误差（波特率）→ 再核数据位 / 校验 / 停止 / 反相（帧格式）→ 长线 RS-485 才看端接和方向时序（信号质量）。**"发 0x55 量波特率""确认 TX-RX 交叉共地"两个动作，永远先做。**

![图10：UART 翻车分层(别一上来换模块/芯片；发 0x55 量波特率+确认交叉共地两个动作先做)。①物理连接层(完全收不到最高频)：TX/RX 没交叉(都 TX 对 TX)/没共地 GND/电平不兼容(TTL 3.3V 直连 RS-232 ±12V 轻则收不到重则烧脚)，收不到先查这层不是调波特率，万用表/示波器确认接对+共地+电平域一致。②波特率/采样层(乱码提速就乱最高频)：两端波特率没对准，发 0x55 量实际算误差超没超±2~3%，9600 通 115200 乱=高波特分频凑不出精确值误差顶穿预算采样点漂隔壁位，对策换能精确分频的波特率/改晶振调分频/降波特。③帧格式层(整片乱码偶发校验错)：数据位/校验/停止两端不一致(8N1 vs 8E1、bit 顺序、反相设反)，核对两端数据位校验停止 LSB first 反相一致。④信号物理质量层(长线 RS-485 才犯)：没端接/端接不对→差分反射振铃误码、DE/RE 方向时机不对截断或挡回复、共模超范围，量差分波形查 120Ω 端接偏置+DE 切换时机(模块一 SI+RS-485 专项)。口诀：查交叉共地电平→发 0x55 量波特率→核帧格式反相→长线看端接方向](/images/53/ill09_debug.png)

## 一句话总结

**一句话：UART 是低速板级总线这条线的第三站、也是最"另类"的一个——它连时钟线都省了，是异步的：收发两端不共享时钟，全靠约定同一个波特率、各自用本地时钟去数去采，判据在"两端约定"而不在统一规范表。**

- **异步、无时钟线，波特率就是命门**：I2C/SPI 有共享时钟兜底，UART 没有——两端波特率必须约成一样、且累计误差压在 **±2%~±3%** 内，否则采样点越数越偏、帧尾先乱。「9600 通、115200 乱」十有八九是**高波特率下分频凑不出精确值**。
- **靠起始位帧同步、帧格式两端必须一致**：线空闲高，**起始位下降沿是发令枪**；一帧 = 起始 + 数据（5~9，通常 8，LSB 先）+ 校验（N/E/O）+ 停止（1/1.5/2）。**8N1 = 10 位**。格式设错（8N1 vs 8E1、bit 顺序、反相）照样乱码。
- **点对点、无主从、无地址、无 ACK**：最基本就是两设备对发，协议极简。要多点，得升级 **RS-485 物理层 + 上层协议（Modbus）加地址和方向控制**。
- **物理层电平多变，接错出大事**：**TTL（0/Vdd）/ RS-232（±12V 反相）/ RS-485（差分、多点、上千米）**。TTL 和 RS-232 绝不能直连（**烧脚**），中间加 MAX232；RS-485 长线高波特要 **120Ω 端接 + 偏置 + DE/RE 方向时序**，模块一 SI 那套回来了。
- **测什么（三类）**：①信号电气（电平门槛 / RS-232 双极性 / RS-485 差分幅度共模端接）；②时序（**发 0x55 量位宽反推波特率、算两端误差**，量帧，RS-485 方向时序）；③协议（UART 解码，设对波特率 / 数据位 / 校验 / 停止 / LSB / 反相，看 framing / parity error）。
- **判据在"约定 + 物理层标准"**：波特率 / 帧格式两端约的必须一样、误差 ≤±2~3%；电平按物理层查（TTL 看手册 V_IH/V_IL、RS-232/485 看 TIA 标准）。**不是统一规范表、不是时序手册、也不是眼图。**
- **调试永远分层**：收不到先查**交叉 / 共地 / 电平**（物理连接）→ 乱码先**发 0x55 量波特率**（波特率）→ 再核**数据位 / 校验 / 停止 / 反相**（帧格式）→ 长线 RS-485 才看**端接和方向时序**（信号质量）。
